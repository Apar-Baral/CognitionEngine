## 📄 EVERY FILE — DETAILED SPECIFICATION

---

### SECTION A: CORE LAYER (The Foundation)

These files are the bedrock. Everything depends on them.

---

#### `src/core/config.py`
**Why it exists:** Centralized configuration prevents chaos. Every component reads from here. No scattered API keys, no hardcoded paths, no magic numbers.

**What it must contain:**
- **Hierarchical config loader:** System defaults → global user config (`~/.cognition/config.yaml`) → project config (`.cognition/config.yaml`) → environment variables → CLI overrides. Each layer overrides the previous.
- **API key manager:** Stores keys for all providers. Never writes keys to project files. Validates keys on load by making a minimal API call (1 token) to check if they work.
- **Model registry path:** Points to a YAML/JSON file that lists all known models dynamically (NOT hardcoded). New models added by editing this file, not code.
- **Default token budgets:** Per session type (BUILD=75000, DEBUG=50000, REFACTOR=60000, EXPLORE=35000, INTEGRATE=55000, OPTIMIZE=30000). Overridable per project.
- **Shield sensitivity:** LOW/MEDIUM/HIGH. Controls validation strictness.
- **Proxy settings:** Port, timeout, retry logic, max concurrent requests.
- **Storage paths:** Where DNA, sessions, truth database, metrics are stored.
- **Visualization preferences:** Color mode, graph style, detail level.
- **Configuration validation:** On load, checks all paths exist, all required keys present, values in valid ranges. Throws clear errors if not.

**Why it's necessary:** Without this, API keys end up in git, paths break when moving machines, and changing a model requires editing 15 files. This is the single source of truth.

---

#### `src/core/constants.py`
**Why it exists:** System-wide immutable definitions. Changes here affect everywhere. Separates "user preferences" from "system rules."

**What it must contain:**
- **Phase status state machine:** Dictionary mapping each status to its valid next states. Example: `{"NOT_STARTED": ["IN_PROGRESS"], "IN_PROGRESS": ["IN_REVIEW", "BLOCKED", "COMPLETED"], "COMPLETED": []}`. Used by phase tracker to validate transitions.
- **Task status state machine:** Same structure for sub-tasks.
- **Session types:** Enum of BUILD, DEBUG, REFACTOR, EXPLORE, INTEGRATE, OPTIMIZE. Each type maps to default token allocation percentages.
- **Hallucination categories:** IMPORT_INVENTION (importing non-existent package), API_INVENTION (calling non-existent method), PARAMETER_INVENTION (wrong function arguments), LOGIC_ERROR (semantically wrong code), DOCUMENTATION_MISMATCH (comment says one thing, code does another).
- **Severity levels:** CRITICAL (would break production), HIGH (would cause bug), MEDIUM (code smell), LOW (style issue), INFO (observation).
- **Budget zones:** GREEN (0-60% consumed: full speed ahead), YELLOW (60-85%: warnings, suggest lighter approaches), RED (85-100%: force lightweight mode, block heavy operations).
- **Supported languages:** List of file extensions → language name mapping for the scanner.
- **Standard library modules per language:** For import validation without requiring requirements.txt for stdlib imports.
- **Vector DB embedding dimensions:** 384 for all-MiniLM, 768 for others. Configurable but defaults set here.
- **Maximum agent count:** Safety limit (default 10) to prevent accidentally spawning 100 agents and burning thousands of dollars.

**Why it's necessary:** Without this, "NOT_STARTED" could transition to anything. Magic strings like "IMPORT_INVENTION" get typoed across files. Budget zones become inconsistent. This is the system's constitution.

---

#### `src/core/exceptions.py`
**Why it exists:** Custom exception hierarchy enables precise error handling. Generic `Exception` catches tell you nothing. Custom exceptions tell you exactly what went wrong and what to do about it.

**What it must contain:**
- **Base exception:** `CognitionEngineError` — all custom exceptions inherit from this.
- **DNA errors:** `DNALoadError` (file missing/corrupt), `DNAValidationError` (schema violation), `DNASaveError` (write failed), `DNAMigrationError` (version migration failed).
- **State transition errors:** `InvalidTransitionError` (attempted impossible transition like COMPLETED→IN_PROGRESS), `TransitionBlockedError` (valid transition but dependencies not met).
- **Shield errors:** `ShieldBlockError` (hallucination confirmed, modification blocked), `ShieldWarningError` (potential issue, modification allowed with warning), `ShieldTimeoutError` (validation took too long).
- **Model errors:** `ModelNotAvailableError` (provider down), `ModelRateLimitError` (rate limited, includes retry_after seconds), `ModelCapabilityError` (model can't do what was asked), `ModelAuthenticationError` (bad API key).
- **Budget errors:** `TokenBudgetExceededError` (session budget hit), `CostThresholdExceededError` (cost limit hit).
- **Agent errors:** `AgentSpawnError` (failed to start agent), `AgentCommunicationError` (inter-agent message failed), `AgentTimeoutError` (agent didn't respond).
- **Bootstrap errors:** `BootstrapGenerationError` (failed to compile context).
- **Planner errors:** `PlanningError` (can't generate plan), `DependencyCycleError` (circular dependency detected).

**Why it's necessary:** Enables code like:
```python
try:
    shield.validate(code)
except ShieldBlockError as e:
    auto_corrector.fix(e.details)
except ShieldWarningError as e:
    log.warning(f"Potential issue: {e.details}")
```
Without custom exceptions, you catch everything and can't differentiate.

#### `src/core/logging_setup.py`
**Why it exists:** Structured logging for debugging distributed systems. When you have 5 agents running in parallel, you need to trace which agent did what, when, and why.

**What it must contain:**
- **Structured JSON logging:** Every log entry is a JSON object with: timestamp (ISO 8601 with microseconds), level, component (which file/module), session_id, agent_id (if in agent context), correlation_id (traces a single user action across all components), message, extra (arbitrary key-value data).
- **Correlation ID propagation:** When the CLI receives a command, it generates a UUID. This UUID is passed to every component that handles that command. Every log entry includes this UUID. Filtering logs by correlation_id shows the complete trace of one user action.
- **Per-component log levels:** Set shield to DEBUG for verbose validation output while keeping proxy at INFO. Controlled via config or environment variables.
- **Session-based log files:** Each session creates a log directory: `data/projects/{name}/sessions/{session_id}/logs/`. Contains `proxy.log`, `shield.log`, `orchestra.log`, `main.log`.
- **Log rotation:** When a log file exceeds 10MB, compress it to `.gz` and start a new file. Keep last 50 session logs, delete older ones.
- **Console output:** Separate from file logging. Console shows INFO and above with colors. Files get DEBUG and above in JSON.
- **Sensitive data redaction:** API keys, tokens, and PII are automatically redacted from logs using regex patterns.

**Why it's necessary:** Without correlation IDs, when something breaks, you can't trace what happened. Without structured logs, you can't query "show me all shield blocks from session 47." Without log rotation, your disk fills up.

#### `src/core/types.py`
**Why it exists:** Shared type definitions used across the entire system. Ensures consistency between components.

**What it must contain:**
- **TypedDict definitions for all major data structures:** SessionData, PhaseData, SubTaskData, AgentConfig, ModelInfo, TokenUsage, HallucinationRecord, InsightRecord, RecommendationRecord, BootstrapContext, SimulationResult, DeviationPlan.
- **Enum definitions:** PhaseStatus, TaskStatus, SessionType, HallucinationCategory, Severity, BudgetZone, ComponentStatus, EdgeType, FeatureType, IntegrationStrategy.
- **Type aliases:** SessionID (str), PhaseID (str), AgentID (str), CorrelationID (str), TokenCount (int), FilePath (str).
- **Union types for common patterns:** ValidationResult (Pass | Block | Warn), AgentResponse (Success | Failure | Timeout).

**Why it's necessary:** Without shared types, component A expects `phase["status"]` as string while component B sets it as enum. Type mismatches cause subtle bugs. This is the contract all components agree on.

#### `src/core/algorithms.py`
**Why it exists:** Centralized algorithm implementations used by multiple components. One implementation, tested once, used everywhere. Optimized for the specific use case.

**What it must contain:**

- **Binary search for sorted session logs:** When looking up a session by timestamp in a chronologically sorted log file, binary search finds it in O(log n) instead of scanning O(n). Used by session_store.py when restoring context from a specific past session.

- **Topological sort for dependency resolution:** Given a list of phases with dependencies, topological sort produces a valid build order. Used by master_planner.py to order phases and dependency_resolver.py to find next executable phase. Algorithm: Kahn's algorithm (BFS-based) for O(V+E) complexity. Detects cycles and raises DependencyCycleError.

- **Dijkstra's shortest path for critical path calculation:** In the phase dependency graph (where edge weights = estimated token cost), Dijkstra's finds the critical path (most expensive dependency chain). Used by critical_path.py to identify the project's bottleneck and estimate minimum completion time.

- **Levenshtein distance for fuzzy matching:** When hallucination shield detects a non-existent function call, it uses Levenshtein distance to find the closest existing function name. `authentcate_v2` → Levenshtein distance 2 from `authenticate` → suggests `authenticate`. Used by auto_corrector.py.

- **Cosine similarity for vector comparisons:** When checking if a proposed code change is similar to existing patterns or when the truth database looks up function signatures. Uses dot product of normalized vectors. Used by truth_database.py and semantic_analyzer.py.

- **Exponential moving average for trend detection:** When knowledge synthesizer tracks hallucination rates over sessions, EMA smooths out noise and detects trends. EMA(rate) = α × current_rate + (1-α) × previous_EMA. α=0.3 gives more weight to recent sessions. Used by trend_analyzer.py.

- **Kalman filter for token prediction:** The budget predictor uses a simple Kalman filter to predict session token consumption. State = [estimated_tokens, token_consumption_rate]. Observation = actual tokens consumed so far. The filter combines the prediction with observation, weighted by uncertainty. More accurate than simple linear extrapolation.

- **Bloom filter for "Do Not Repeat" register:** The avoid register needs to quickly check "has this file already been marked as understood?" without storing all filenames in memory. Bloom filter gives O(1) lookup with configurable false positive rate (1% acceptable). Used by avoid_register.py.

- **LRU cache with TTL for DNA queries:** Strategic memory queries the DNA frequently. An LRU (Least Recently Used) cache with Time-To-Live stores recent query results. Cache key = query hash, value = result. Max 1000 entries, TTL 30 seconds. Used by strategic_memory.py.

**Why it's necessary:** Algorithms implemented once, correctly, with known complexity. No duplicated buggy implementations across files. When a component needs fuzzy matching, it imports `from core.algorithms import levenshtein_distance` — not copy-paste from Stack Overflow.

### SECTION B: SCANNER LAYER (Understanding the Project)

These files analyze the existing codebase to understand what's being built.

---

#### `src/scanner/project_scanner.py`
**Why it exists:** Before the system can plan, it must understand the existing project. This is the entry point for codebase analysis.

**What it must contain:**
- **Directory traversal:** Walks the project tree. Respects `.gitignore` patterns. Skips `node_modules`, `__pycache__`, `.venv`, `dist`, `build` directories.
- **File classification:** Categorizes every file: SOURCE_CODE, CONFIG, DOCUMENTATION, TEST, ASSET, UNKNOWN. Based on extension and location.
- **Project structure summary:** Counts files per type, directories per type, total lines of code (using language-specific comment stripping).
- **Entry point detection:** Identifies `main.py`, `app.py`, `index.js`, `main.go`, etc. These are the project's entry points.
- **Existing Claude Code configuration detection:** Checks for `CLAUDE.md`, `.claude/` directory, existing hooks. Warns if conflicts might occur.
- **Git repository analysis:** If git repo, extracts: last commit date, total commits, active branches, contributors. This helps the planner understand project maturity.
- **Output:** A `ProjectScanResult` object containing all discovered information. This is the input to the master planner.

**Why it's necessary:** The planner can't plan without knowing what already exists. You can't add OAuth to a project if you don't know it already has authentication.

---

#### `src/scanner/language_detector.py`
**Why it exists:** Multi-language support. Detects the primary and secondary languages of the project.

**What it must contain:**
- **Extension-based detection:** Maps file extensions to languages (.py→Python, .js/.ts→JavaScript/TypeScript, .go→Go, .rs→Rust, .java→Java, .rb→Ruby).
- **Framework-specific file detection:** Detects `requirements.txt`, `pyproject.toml` → Python project. `package.json` → Node.js. `go.mod` → Go. `Cargo.toml` → Rust. `Gemfile` → Ruby.
- **Language percentage breakdown:** "73% Python, 15% JavaScript, 8% HTML, 4% CSS". This determines which validators and parsers to activate.
- **Type system detection:** For Python, detects if using type hints. For JavaScript, detects TypeScript vs plain JS. This affects static analysis depth.
- **Output:** A `LanguageProfile` listing primary language, all detected languages with percentages, and framework hints.

**Why it's necessary:** The hallucination shield needs to know which AST parser to use. The planner needs to know which conventions to apply. You can't validate Python code with a JavaScript parser.

---

#### `src/scanner/framework_detector.py`
**Why it exists:** Detects which frameworks and libraries are in use. Critical for accurate hallucination detection.

**What it must contain:**
- **Import analysis:** Scans source files for imports. Builds a set of all imported packages.
- **Requirements file parsing:** Parses `requirements.txt` (with version specifiers), `package.json` (dependencies + devDependencies), `go.mod`, `Cargo.toml`.
- **Framework identification:** Maps packages to known frameworks. `flask`/`django`/`fastapi` → web framework. `sqlalchemy`/`prisma` → ORM. `pytest`/`jest` → test framework. `react`/`vue`/`angular` → frontend.
- **Version extraction:** Gets installed versions (from lock files if available). This matters because APIs change between versions.
- **Output:** A `FrameworkProfile` listing all detected packages, their versions, and categorized frameworks.

**Why it's necessary:** When the shield validates `from flask_sqlalchemy import SQLAlchemy`, it needs to know: (a) is flask_sqlalchemy in requirements.txt? (b) is the installed version compatible with this import style? Without this, valid imports get flagged as hallucinations.

---

#### `src/scanner/dependency_parser.py`
**Why it exists:** Builds a dependency graph of the project's internal modules. Which file imports which other file.

**What it must contain:**
- **Import extraction:** Parses all source files to extract import statements. Records: importing_file, imported_module, imported_symbols.
- **Module resolution:** Resolves relative imports to absolute file paths. `from ..utils.helpers import hash_password` → `src/utils/helpers.py`.
- **Dependency graph construction:** Builds a directed graph where nodes are files and edges are imports. Tracks: which files are most imported (high risk if modified), which files import the most (complex, likely god objects).
- **Circular dependency detection:** Uses DFS to detect cycles. Circular dependencies indicate architectural problems and are flagged for the planner.
- **Unused import detection:** Imports that exist but are never used. Flagged as technical debt.
- **Output:** A `DependencyGraph` object used by the master planner, the architecture graph, and the truth database.

**Why it's necessary:** When the planner breaks the project into phases, it needs to know that modifying `database.py` affects 15 other files. When the shield validates a change, it needs to know all files that might be affected.

---

#### `src/scanner/codebase_indexer.py`
**Why it exists:** Creates the initial truth database. Indexes every function, class, method, and API endpoint in the codebase so the hallucination shield can validate against reality.

**What it must contain:**
- **AST-based extraction:** Parses every source file into an AST. Walks the tree to extract: function definitions (name, parameters, return type hints, docstring), class definitions (name, methods, attributes), variable assignments (name, type hint), API route definitions (path, HTTP method, handler function).
- **Vector embedding generation:** For each extracted symbol, generates an embedding using a lightweight model (all-MiniLM-L6-v2). This enables semantic search: "find the function that handles user authentication" even if it's named `process_login`.
- **Cross-referencing:** Links each function call to its definition. `authenticate()` called in `login.py` → defined in `auth/utils.py`. This enables "go to definition" style validation.
- **Incremental update support:** After initial indexing, only re-index files that changed. Uses file modification timestamps and content hashes.
- **Index storage:** Stores the index in the truth database (vector store). Also maintains a quick-lookup hash map: function_name → definition_location.
- **Output:** Populates the `data/projects/{name}/truth_db/` directory with the initial index.

**Why it's necessary:** The hallucination shield's core capability is answering "does this function exist?" This index is the source of that answer. Without it, every function call is assumed valid until proven otherwise.

### SECTION C: PARSER LAYER (Understanding User Intent)

These files interpret what the user wants to do.

---

#### `src/parser/intent_parser.py`
**Why it exists:** Translates natural language user requests into structured actions. "Add OAuth login" isn't a command — it's intent that needs parsing.

**What it must contain:**
- **Intent classification:** Classifies user input into: CREATE_FEATURE, FIX_BUG, REFACTOR_CODE, EXPLORE_CODEBASE, ADD_TESTS, OPTIMIZE_PERFORMANCE, UPDATE_DEPENDENCY, ASK_QUESTION, UNKNOWN.
- **Entity extraction:** Extracts entities from the request. "Add OAuth login with Google and GitHub" → feature="OAuth login", providers=["Google", "GitHub"], scope="authentication".
- **Context awareness:** Considers the current project state. "Fix the login bug" → checks recent sessions for known bugs → checks technical debt register for login-related items.
- **Ambiguity resolution:** "Make it faster" → with no context → asks "What specifically? API response time? Database queries? Build time?" rather than guessing.
- **Confidence scoring:** Every parsed intent has a confidence score. Low confidence (<0.7) triggers a clarification prompt. High confidence (>0.9) proceeds automatically.
- **Output:** An `Intent` object with: action_type, entities (dict), confidence, clarification_needed (bool), suggested_clarification (str if needed).

**Why it's necessary:** The master planner can't work with "make it better." It needs structured data. This bridges natural language → machine-executable plans.

---

#### `src/parser/task_extractor.py`
**Why it exists:** Breaks a parsed intent into discrete tasks. "Add OAuth login" isn't one task — it's 8-15 sub-tasks.

**What it must contain:**
- **Task decomposition heuristics:** For a CREATE_FEATURE intent, generates tasks like: [research best practices, design data model, implement backend, implement frontend, write tests, update documentation, security review, integration test].
- **Project-specific customization:** If the scanner detected Flask, tasks include Flask-specific steps. If React detected, includes component tasks.
- **Task ordering:** Orders tasks by dependency. "Design data model" before "Implement backend." "Write tests" after "Implement backend" but "Write unit tests" can be parallel with "Implement frontend."
- **Effort estimation:** Estimates token consumption per task based on: task type, project size, historical data from similar tasks in past sessions.
- **Output:** An ordered list of `Task` objects, each with: description, estimated_tokens, dependencies, parallelizable (bool), suggested_agent_type.

**Why it's necessary:** The planner assigns tasks to phases. The orchestrator assigns tasks to agents. Without structured tasks, both components guess.

---

#### `src/parser/scope_detector.py`
**Why it exists:** Determines the scope of a request. Is the user asking to modify one file, one module, or the entire project?

**What it must contain:**
- **File-level scope detection:** "Fix the typo in utils.py line 42" → scope is single file, single line.
- **Module-level scope detection:** "Refactor the authentication module" → scope is `src/auth/` directory and all its imports.
- **Project-wide scope detection:** "Add TypeScript support" → scope is entire project.
- **Impact radius estimation:** Given a change, estimates how many files will be affected. Uses the dependency graph from scanner. "Change database.py" → 15 files import it → impact radius = 15.
- **Scope creep detection:** During a session, if the agent starts modifying files outside the original scope, detects and flags. "You were fixing the login bug but you're now modifying the payment system. Is this intentional?"
- **Output:** A `Scope` object with: scope_level (FILE/MODULE/PROJECT), affected_files (list), affected_modules (list), estimated_impact_radius.

**Why it's necessary:** Prevents "fix one bug, create three more." When the orchestrator assigns work to agents, it uses scope to limit what each agent can touch.

### SECTION D: PLANNER LAYER (The Brain)

These files create the master plan and adapt it when things change.

---

#### `src/planner/master_planner.py`
**Why it exists:** This is the core intelligence. It takes the scanned project + user intent and generates the complete 20-30 phase master plan.

**What it must contain:**
- **Phase generation algorithm:** Takes the project goal (from intent parser) and the codebase analysis (from scanner). Generates phases using a dependency-aware decomposition:
  1. Identify the end state: what does "done" look like?
  2. Work backwards: what needs to exist before that?
  3. Group related work into phases (aim for 20-30 phases, each 2-5 sessions)
  4. Order by dependency chain
  5. Assign phase types (BUILD/INTEGRATE/REFACTOR/TEST)
  6. Estimate tokens per phase
- **Phase metadata assignment:** Each phase gets: id (PHASE_01), name, description, phase_type, estimated_tokens, dependencies (list of phase_ids), sub_tasks (from task_extractor), priority.
- **Critical path calculation:** Uses the algorithms module's topological sort + Dijkstra's to find the critical path. This is the minimum viable sequence.
- **Parallelism identification:** Phases that don't depend on each other are marked as parallelizable. The orchestrator can run these simultaneously.
- **Checkpoint definition:** Every 5 phases, a checkpoint phase is inserted: REVIEW_AND_TEST. Forces validation before proceeding.
- **Output:** A complete `MasterPlan` object written to `dna.json`. This is the project's roadmap.

**Why it's necessary:** Without this, the system has no idea what to do next. Every session would require the user to manually specify tasks. This is the difference between "AI assistant" and "autonomous development partner."

---

#### `src/planner/phase_generator.py`
**Why it exists:** Generates individual phases with proper granularity. Not too big (impossible to complete) and not too small (administrative overhead).

**What it must contain:**
- **Granularity calibration:** Each phase should consume 2-5 sessions (roughly 100k-400k tokens). If estimated >500k tokens, split into two phases. If <50k tokens, merge with adjacent phase.
- **Phase naming convention:** Descriptive and consistent. "Implement User Authentication Backend" not "Auth stuff."
- **Sub-task generation:** Each phase contains 3-8 sub-tasks. Sub-tasks are the atomic units the orchestrator assigns to agents.
- **Deliverable definition:** Every phase has clear deliverables: what files will be created/modified, what tests will pass, what documentation will exist.
- **Completion criteria:** Objective, testable criteria. "All 12 unit tests pass" not "authentication works well."
- **Template-based generation:** Uses templates per phase type: BUILD phases have implementation sub-tasks, TEST phases have test-writing sub-tasks, REFACTOR phases have analysis+implementation sub-tasks.

**Why it's necessary:** Inconsistent granularity leads to frustration — phases that take forever or phases that feel like busywork. This ensures every phase is a meaningful unit of progress.

---

#### `src/planner/dependency_graph.py`
**Why it exists:** Manages the dependency relationships between phases. This determines execution order.

**What it must contain:**
- **Graph construction:** Builds a Directed Acyclic Graph (DAG) where nodes are phases and edges are dependencies. Edge from A to B means "A must complete before B starts."
- **Cycle detection:** Uses DFS-based cycle detection. If a cycle is found, it's reported clearly: "PHASE_05 depends on PHASE_08 which depends on PHASE_05. Break the cycle by..."
- **Transitive reduction:** Simplifies the graph. If A→B→C, the edge A→C is redundant and removed. This makes the graph cleaner.
- **Topological ordering:** Produces a valid execution order. Multiple valid orders may exist; the one that maximizes parallelism is preferred.
- **Dependency type classification:** HARD dependency (must complete before), SOFT dependency (should complete before, but not blocking), PARALLEL (can run simultaneously).
- **Impact analysis:** "If PHASE_04 is delayed, which phases are affected?" Traverses forward from the delayed phase to find all dependents.

**Why it's necessary:** Without this, phases execute in random order. You'd try to implement the frontend before the API exists. This is the project's skeleton.

---

#### `src/planner/subtask_decomposer.py`
**Why it exists:** Breaks phases into atomic sub-tasks that can be assigned to individual agents.

**What it must contain:**
- **Decomposition by phase type:** BUILD phases → [design component, implement core logic, add error handling, write unit tests, document API]. REFACTOR phases → [analyze current code, identify improvements, implement changes, verify behavior unchanged, update tests].
- **Task sizing:** Each sub-task should be completable in one agent session (20k-60k tokens). If larger, split further.
- **Parallelism annotation:** Marks which sub-tasks can run in parallel. "Write unit tests for module A" and "Write unit tests for module B" can run simultaneously if modules A and B are independent.
- **Agent type assignment:** Suggests which agent type should handle each sub-task: ARCHITECT for design tasks, BACKEND_DEV for implementation, SECURITY_REVIEWER for security tasks, TEST_WRITER for test tasks.
- **Input/output specification:** Each sub-task specifies: required context (which files/information needed), expected output (new files, modified files, test results).

**Why it's necessary:** The orchestrator needs atomic work units. "Build authentication" is too vague. "Implement password hashing function in auth/utils.py" is executable.

---

#### `src/planner/budget_assigner.py`
**Why it exists:** Assigns token budgets to phases and sub-tasks based on historical data and complexity estimation.

**What it must contain:**
- **Historical budget lookup:** Queries the cross-project transfer module for similar phases from past projects. "BUILD phase for authentication module in Python Flask project: historically 120k-180k tokens."
- **Complexity-based estimation:** Factors: number of files affected, number of new files, lines of code estimated, test coverage requirements, integration complexity.
- **Budget buffer:** Adds 20% buffer to estimates for unexpected complexity. A 100k estimate becomes 120k budget.
- **Per-phase allocation:** Splits the phase budget across sub-tasks: implementation gets 50%, testing gets 25%, documentation gets 10%, review/buffer gets 15%.
- **Dynamic adjustment:** After each session, compares actual vs estimated. If consistently underestimating, increases buffer. If consistently overestimating, decreases. Uses the Kalman filter from algorithms.py.
- **Output:** Every phase and sub-task gets an `estimated_tokens` field and a `budget_tokens` field (estimate + buffer).

**Why it's necessary:** Without budgets, sessions burn tokens without accountability. With budgets, the Token Guardian can enforce limits and the user knows what to expect.

### SECTION E: DNA LAYER (Persistent Project Knowledge)

These files manage the Project DNA — the single source of truth that persists across all sessions.

---

#### `src/dna/schema.py`
**Why it exists:** Defines the exact structure of the project DNA JSON file. This is the contract — every component reads and writes according to this schema.

**What it must contain:**
- **Complete JSON Schema (draft-07):** Defines every field, its type, whether it's required, valid values, constraints (min/max, regex patterns).
- **Top-level structure:** schema_version, project (metadata), master_plan (all phases), architecture_graph (nodes + edges), feature_registry (planned + emergent features), deviation_history (all deviations), avoid_registry (hallucinations, understood files, failed approaches), insights (generated insights), recommendations (active recommendations), rl_state (Q-table for token allocation), sessions_index (quick lookup for all sessions).
- **Phase schema:** id (PHASE_XX format), name, description, status (enum), completion_score (0-100), started, completed, sessions_used, tokens_consumed, estimated_tokens, budget_tokens, deliverables (list), dependencies (list), blocked_by (list), sub_tasks (array of sub-task objects), state_history (array of state transitions), phase_type (enum), insights_generated (array).
- **Sub-task schema:** id, name, status (enum), progress (0-100), assigned_agent, last_worked_on, estimated_tokens, actual_tokens, files_modified (list), next_action, completion_criteria.
- **Architecture node schema:** id, type (enum), status, description, files (list), interfaces (list), dependencies (list), created_in_phase, last_modified_in_session.
- **Architecture edge schema:** source, target, type (enum), description, created_in_phase.
- **Feature schema (planned):** id, name, description, priority (1-5), estimated_tokens, dependencies, phase_id, status.
- **Feature schema (emergent):** id, name, description, discovered_in_session, integration_strategy, disruption_score, status, phases_created, phases_modified.
- **Insight schema:** id, type, finding, confidence, generated_at, session_id, actionability, applied, impact_if_applied.
- **Recommendation schema:** id, type, description, priority, estimated_impact, generated_at, accepted.

**Why it's necessary:** Without a schema, the DNA becomes a garbage dump of inconsistent data. One component writes `status: "in_progress"`, another writes `status: "IN_PROGRESS"`. Validation catches this immediately.

---

#### `src/dna/loader.py`
**Why it exists:** Handles all DNA file I/O with safety guarantees: atomic writes, automatic backups, corruption recovery.

**What it must contain:**
- **Atomic write mechanism:** Write to temp file → validate temp file → rename to actual path. If crash occurs during write, original file is untouched.
- **Automatic backup:** Before every save, create timestamped backup in `data/projects/{name}/backups/`. Keep last 20 backups, delete older ones.
- **Corruption recovery:** On load failure, automatically try the most recent backup. If that fails, try the next. If all backups fail, raise DNALoadError with clear recovery instructions.
- **In-memory caching:** After loading, cache the DNA in memory with a 30-second TTL. Subsequent reads in the same session hit cache, not disk. Force reload available for multi-process scenarios.
- **Schema validation on load:** Every load validates against the JSON schema. Invalid DNA raises validation errors — no silent corruption.
- **Version migration:** If schema_version in DNA is older than current, automatically runs migration functions to update the structure.
- **Concurrent access protection:** File locking (fcntl on Unix) prevents two processes from writing simultaneously.

**Why it's necessary:** The DNA is the system's memory. If it gets corrupted, the system gets amnesia. Atomic writes and backups make corruption nearly impossible.

---

#### `src/dna/validator.py`
**Why it exists:** Validates DNA against the schema on every load and save. Catches data corruption before it propagates.

**What it must contain:**
- **Schema validation:** Uses jsonschema library to validate against the schema defined in schema.py.
- **Semantic validation (beyond schema):** Checks that dependencies reference real phase IDs. Checks that phase IDs are sequential and complete (no gaps). Checks that architecture edges reference real node IDs. Checks that session counts match actual session logs.
- **Consistency checks:** Total tokens consumed should equal sum of all phase tokens. Phase completion percentage should match sub-task completion. Current phase index should point to a real phase.
- **Warning generation:** Non-fatal issues generate warnings: "Phase 12 has no sub-tasks defined" — valid JSON but semantically incomplete.
- **Repair suggestions:** For detected issues, suggests fixes: "PHASE_05 references dependency PHASE_15 which doesn't exist. Did you mean PHASE_05?"
- **Validation report:** Returns a structured report with: is_valid (bool), errors (list, each with path, message, severity), warnings (list).

**Why it's necessary:** A schema validates structure. This validates meaning. "Phase 5" existing in JSON doesn't mean it's correct.

---

#### `src/dna/mutator.py`
**Why it exists:** Provides safe, controlled mutations to the DNA. No component modifies DNA directly — they go through the mutator.

**What it must contain:**
- **Atomic mutation operations:** update_phase_status(), add_session_record(), add_hallucination_to_avoid_registry(), add_insight(), update_architecture_node(), add_emergent_feature(), update_sub_task_progress().
- **Pre-mutation validation:** Before applying mutation, validates that the resulting DNA would be valid. If mutation would corrupt DNA, it's rejected.
- **Audit trail:** Every mutation logs: who (which component), what (mutation type), when (timestamp), before_value, after_value. Stored in the DNA's mutation_history.
- **Batch mutations:** Multiple changes applied atomically. All succeed or all fail.
- **Conflict detection:** If two components try to mutate the same field, the second one gets a conflict error with the current value.
- **Rollback support:** If a mutation fails mid-way, the DNA is restored to pre-mutation state.

**Why it's necessary:** Without controlled access, any component can break the DNA. With the mutator, all changes are validated, logged, and recoverable.

#### `src/dna/query.py`
**Why it exists:** Provides a clean query interface for the DNA. Components don't dig through raw JSON — they ask questions.

**What it must contain:**
- **Phase queries:** get_current_phase(), get_phase_by_id(), get_phases_by_status(), get_next_executable_phase(), get_blocked_phases(), get_completed_phases().
- **Task queries:** get_active_sub_tasks(), get_sub_tasks_for_phase(), get_next_pending_sub_task().
- **Dependency queries:** what_depends_on(phase_id), what_does_phase_depend_on(phase_id), find_blockers_for_phase(phase_id).
- **Architecture queries:** get_operational_components(), get_component_dependencies(), find_components_affected_by_file(file_path).
- **Feature queries:** get_planned_features(), get_emergent_features(), get_features_by_priority(), find_unintegrated_emergent_features().
- **Insight queries:** get_unapplied_insights(), get_insights_by_type(), get_high_confidence_insights().
- **Session queries:** get_sessions_for_phase(), get_last_session(), get_session_by_id(), get_sessions_in_date_range().
- **Aggregation queries:** calculate_project_completion(), estimate_remaining_tokens(), get_total_cost(), get_efficiency_trend().

**Why it's necessary:** If the DNA structure changes (migration), only query.py needs updating. All components using query.py continue working. This is the abstraction layer.

#### `src/dna/migrations.py`
**Why it exists:** DNA schema will evolve. Migrations allow old DNA files to be upgraded without data loss.

**What it must contain:**
- **Version tracking:** Current schema version stored as constant. DNA files store their version.
- **Migration functions:** Each migration is a function that transforms DNA from version X to X+1. Example: v1.0→v1.1 adds `phase_type` field to phases, defaulting to "BUILD".
- **Migration chain:** System detects current version, runs all migrations sequentially until up to date.
- **Migration testing:** Each migration has a test with sample old-version DNA, runs migration, validates output.
- **Rollback information:** Each migration stores what it changed, enabling manual rollback if needed.
- **Backup before migration:** Before running migrations, a backup is created with the old version.

**Why it's necessary:** Without migrations, adding a new field to the schema breaks all existing projects. With migrations, old projects upgrade seamlessly.

### SECTION F: MEMORY LAYER (Three-Tier Memory System)

These files maintain the system's awareness across time. Without them, every session starts blind.

---

#### `src/memory/strategic_memory.py`
**Why it exists:** Permanent, project-level memory. Answers "where are we in the 24-phase plan?" Survives forever. Updated after every session.

**What it must contain:**
- **Phase state management:** Maintains the current state of all phases. When Phase 7 completes, Phase 8 auto-activates. When a phase blocks, all dependent phases are flagged.
- **Dependency resolution engine:** Given any phase ID, traverses the dependency graph to determine if it's executable (all dependencies met). Uses the topological sort from algorithms.py.
- **Critical path tracker:** Maintains the current critical path. When phases complete or delay, recalculates. Shows "the fastest this project can complete is 18 more phases, estimated 45 sessions."
- **Blockage detection:** Proactively scans for potential blockers. "PHASE_12 will be blocked in 3 sessions because PHASE_09 is falling behind schedule."
- **Completion metrics:** Overall project completion percentage. Phase-by-phase breakdown. Velocity (phases completed per week). Burn rate (tokens consumed per phase vs estimated).
- **Architecture graph management:** Maintains the live architecture graph. As phases complete, nodes transition: planned → in_development → operational. Edges are added as integrations happen.
- **Feature registry management:** Tracks all features (planned and emergent). When a feature is fully integrated, its status updates. The registry is the system's answer to "what are we building?"
- **Snapshot creation:** Before major changes (new phase, deviation), creates a snapshot of the current DNA state. Enables "go back to before we added that feature."

**Why it's necessary:** This is the system's long-term memory. Without it, every session asks "what are we building?" This provides instant, complete project awareness.

---

#### `src/memory/tactical_memory.py`
**Why it exists:** Per-phase memory. Detailed awareness of the current phase's state. Only needs to track 1-2 active phases at a time.

**What it must contain:**
- **Active sub-task tracking:** Which sub-tasks are in progress, which are done, which are blocked. Progress percentage per sub-task.
- **File-level awareness:** Which files are being modified in this phase, which files have been "understood" (don't re-read), which files are causing issues.
- **Agent assignment tracking:** Which agent is working on which sub-task. Agent performance per sub-task (tokens used, efficiency, hallucinations generated).
- **Phase-specific avoid register:** Errors made in this phase that shouldn't be repeated. "Don't import from auth.v1 — use auth.v2." More granular than the global avoid register.
- **Context compression for phase:** When bootstrapping, tactical memory provides the phase-specific context: "You're working on PHASE_07. Sub-task P7_T3 is active. These 4 files are relevant. This is what the last session completed."
- **Phase completion predictor:** Based on sub-task completion rates, predicts when this phase will complete. "At current velocity, Phase 07 will complete in 2 sessions (4 days)."
- **Scope enforcement:** Ensures agents working on this phase don't drift into other phases' territory. "You're modifying a file owned by PHASE_12. Authorized?"

**Why it's necessary:** Strategic memory knows the whole war. Tactical memory knows the current battle. This separation prevents information overload — the bootstrapper gets global context from strategic memory and detailed context from tactical memory.

---

#### `src/memory/operational_memory.py`
**Why it exists:** Per-session memory. The most granular level. Only lasts for one session, but feeds into tactical and strategic memory after session end.

**What it must contain:**
- **Real-time session state:** Current session ID, start time, token consumption so far, budget remaining, agent statuses, files currently being edited.
- **API call log:** Every API call made in this session: timestamp, model used, tokens consumed (input + output), purpose (which agent, which sub-task), latency, success/failure.
- **File modification journal:** Every file read, every file written. Tracks re-reads for the re-read tax calculation. Tracks modifications for the scope creep detector.
- **Hallucination log:** Every hallucination caught: what was proposed, what was wrong, what the correction was, stage that caught it (1/2/3).
- **Decision log:** Every significant decision made: architectural choices, library selections, approach changes. Why was this decision made? What were the alternatives?
- **User interaction log:** Every time the system asked the user a question and the response. Provides context for future sessions.
- **Performance metrics:** Response times, token efficiency per action, agent utilization percentages.
- **Session-end handoff:** When session ends, operational memory compresses into a session summary, adds to session log, and feeds insights to the knowledge synthesizer.

**Why it's necessary:** This is the raw data that all learning depends on. Without detailed operational memory, the knowledge synthesizer has nothing to analyze. Every session generates rich data for future optimization.

---

#### `src/memory/session_store.py`
**Why it exists:** Persistent storage and retrieval of session logs. The database of everything that happened.

**What it must contain:**
- **Session log writer:** Writes operational memory to disk as structured JSON Lines files. One line per significant event. Efficient for append-only writes.
- **Session log reader:** Reads past sessions for bootstrapping, analysis, and visualization. Supports filtering by: session ID range, date range, phase, agent, hallucination type.
- **Binary search for time-based lookup:** When the system needs "what happened in the session from 3 days ago?", binary search on chronologically sorted log lines finds it in O(log n).
- **Session index maintenance:** Maintains an index file mapping session IDs to file offsets for instant lookup. Updated after each session end.
- **Log compression:** Sessions older than 30 days are automatically compressed (gzip). The index tracks which sessions are compressed and decompresses on demand.
- **Cross-session query support:** "Show me all sessions where hallucination rate exceeded 5%." Scans session logs, returns matching session IDs.
- **Storage management:** Enforces retention policies. Keeps last 90 days of detailed logs. Summarizes older sessions (keep stats, discard detailed event log). Prevents disk from filling.

**Why it's necessary:** Without persistent session storage, every session is an island. With it, the system can answer "have we encountered this bug before?" and "which approach worked best last time?"

---

#### `src/memory/metrics_store.py`
**Why it exists:** Time-series metrics database for trends, dashboards, and optimization. Complements session logs with aggregated data.

**What it must contain:**
- **Time-series data structure:** Each metric is (timestamp, metric_name, value, tags). Tags enable multi-dimensional queries: "token_efficiency for agent_type=BACKEND_DEV during phase_type=BUILD."
- **Metrics collected:** tokens_per_session, tokens_per_phase, tokens_per_agent, hallucination_rate, efficiency_score, re_read_tax, budget_adherence (actual/estimated), agent_utilization, phase_velocity, file_count, code_complexity_score, test_coverage_estimate.
- **Aggregation windows:** Raw data at per-minute resolution. Automatic rollup to hourly (kept 90 days), daily (kept 1 year), weekly (kept forever).
- **Trend detection:** Exponential moving average on all metrics. Flags when a metric deviates from its EMA by more than 2 standard deviations. "Hallucination rate spiking — was 2% now 8%."
- **Anomaly detection:** Simple statistical anomaly detection on token consumption. If a session consumes 3x the predicted tokens, it's flagged for investigation.
- **Export capabilities:** Export metrics to CSV for external analysis. Export dashboard data as JSON for visualization.
- **Storage efficiency:** Uses SQLite for storage. Indexed by (timestamp, metric_name). Query performance: <10ms for typical dashboard queries.

**Why it's necessary:** Session logs answer "what happened?" Metrics answer "how are we doing?" and "are we getting better?" The dashboard and RL optimizer depend on this data.

### SECTION G: BOOTSTRAP LAYER (Session Initialization)

These files prepare the perfect context for each new session. This is what eliminates the "figuring out where I left off" time.

---

#### `src/bootstrap/bootstrap_generator.py`
**Why it exists:** The main orchestrator for session preparation. Takes all memory sources, compiles them into an optimized initial context for the AI model.

**What it must contain:**
- **Multi-source context assembly:** Pulls from strategic memory (project overview, current phase), tactical memory (active sub-tasks, relevant files), operational memory (last session summary, what was unfinished), avoid register (don't repeat these mistakes), future you protocol (message from past self).
- **Token budget calculation:** Queries budget_predictor for estimated tokens needed. Applies the RL optimizer's recommended allocation split. Sets the budget for this session.
- **Context prioritization:** Not all context is equally important. Ranks context items by relevance: MUST_INCLUDE (current sub-task, last session decisions), SHOULD_INCLUDE (phase overview, relevant files), NICE_TO_HAVE (project history, related completed phases), EXCLUDE (irrelevant phases, old decisions).
- **Compression strategy:** If total context exceeds 2000 tokens (the sweet spot for initial prompts), compresses lower-priority items using summarization. MUST_INCLUDE items are never compressed.
- **Model-specific formatting:** Different models have different system prompt formats. Adapts the bootstrap to match: Claude's system parameter, OpenAI's system message, DeepSeek's prefix, etc.
- **Bootstrap injection:** Writes the compiled bootstrap to the appropriate location: CLAUDE.md for Claude Code, system prompt for direct API calls, or initial message for chat interfaces.
- **Validation:** After generation, validates that the bootstrap doesn't exceed the budget, contains all MUST_INCLUDE items, and is parseable by the target model.

**Why it's necessary:** This is the difference between "Claude Code, help me with my project" (vague, wastes 10k tokens on exploration) and "Claude Code, continue PHASE_07 sub-task P7_T3, here's exactly what's relevant" (focused, immediate productivity).

---

#### `src/bootstrap/context_compiler.py`
**Why it exists:** Compiles raw project data into token-efficient context blocks. Turns 50 files into 5 relevant summaries.

**What it must contain:**
- **File relevance scoring:** For each file in the project, scores relevance to the current task using cosine similarity between task description embedding and file content embedding. Files scoring >0.4 are "relevant" and included. Files scoring 0.2-0.4 get a one-line summary. Files scoring <0.2 are excluded.
- **Code summarization:** For large files (>500 lines), generates a compressed representation: class/function signatures, docstrings, important constants. Not the full implementation. Reduces a 2000-line file to 200 tokens.
- **Import graph summarization:** "This module depends on: database.py, auth.py, utils.py. These 3 modules depend on this module." Contextualizes the file without listing every import.
- **Change diff integration:** If the last session modified files, includes the diff (not the full file). Shows what changed, not what stayed the same.
- **Conversation history compression:** Summarizes the previous session's conversation into structured bullets: decisions made, approaches rejected, insights gained. Not the verbatim chat log.
- **Token counting during compilation:** Tracks token consumption as it compiles. If approaching the 2000-token limit for the bootstrap, switches to more aggressive compression.
- **Template system:** Uses templates for different session types. BUILD session template emphasizes architecture and existing code patterns. DEBUG session template emphasizes error logs and recent changes.

**Why it's necessary:** Raw context is bloated. A 50-file project has 50,000+ tokens of code. The AI needs maybe 2,000 tokens of well-chosen context. This bridge between "everything" and "exactly what matters."

---

#### `src/bootstrap/budget_predictor.py`
**Why it exists:** Predicts how many tokens a session will consume before it starts. Enables informed decisions: "this task will cost $3. Should I proceed?"

**What it must contain:**
- **Kalman filter prediction model:** Uses a Kalman filter (from algorithms.py) with state = [estimated_total_tokens, token_consumption_rate]. Observation = actual tokens consumed so far in similar tasks. The filter combines prediction (based on sub-task type and project size) with observation (historical data from similar sessions) weighted by uncertainty.
- **Feature extraction for prediction:** Extracts features from the sub-task: phase_type, sub_task_type, estimated_files_affected, estimated_lines_changed, language, framework, agent_count. These features feed into the prediction.
- **Historical baseline:** Queries metrics_store for similar sessions: same phase_type, same language, similar files_affected. Returns the distribution of actual token consumption.
- **Prediction confidence interval:** Outputs not just a point estimate but a range: "Predicted: 48,500 tokens (90% confidence interval: 38,000 - 62,000)." The Kalman filter naturally provides uncertainty.
- **Cost projection:** Converts token prediction to cost using current pricing for the assigned model. "Estimated cost: $1.21 (Claude Sonnet) or $0.18 (DeepSeek)."
- **Budget recommendation:** Recommends a budget = predicted_tokens * 1.2 (20% buffer). If historical accuracy is high (prediction within 10% for last 10 sessions), reduces buffer to 10%.
- **Post-session calibration:** After session ends, compares prediction to actual. Updates the Kalman filter's uncertainty parameters. The predictor gets better with every session.

**Why it's necessary:** Without prediction, every session is a financial surprise. With prediction, you can decide "this refactor isn't worth 200k tokens right now." This is the difference between "using AI" and "managing AI costs."

---

#### `src/bootstrap/avoid_register.py`
**Why it exists:** Maintains the "Do Not Repeat" knowledge base. Prevents the AI from making the same mistake twice.

**What it must contain:**
- **Hallucination registry:** Every hallucination caught by the shield is recorded: what was proposed, what was wrong, what's correct, file context, phase context. Before each session, the bootstrap includes the top 5 most relevant avoid items.
- **Bloom filter for fast lookup:** Uses a Bloom filter (from algorithms.py) for O(1) "have we seen this hallucination before?" checks. Configurable 1% false positive rate. When a hallucination occurs, the bloom filter is checked — if already present, it's a repeating pattern and gets higher priority in the bootstrap.
- **Understood files registry:** Files that have been thoroughly analyzed and don't need re-reading. The bootstrap tells the AI: "These 8 files are fully understood — don't re-read them." Reduces re-read tax.
- **Failed approaches registry:** Approaches that were tried and failed. "We tried using async SQLAlchemy for this — it caused connection pool exhaustion. Use sync SQLAlchemy with background tasks instead." Prevents re-exploring dead ends.
- **Deprecated patterns registry:** Patterns that were used but are being phased out. "We're migrating from `auth.check_token()` to `auth.verify_token()`. Use the new one."
- **Contextual relevance scoring:** When bootstrapping, avoid items are scored for relevance to the current task. Items about authentication aren't shown when working on the payment module.
- **Decay mechanism:** Avoid items that haven't been relevant for 30 sessions are archived (not deleted, just not actively shown). Prevents the avoid register from becoming a dumping ground.

**Why it's necessary:** AI models don't remember their mistakes across sessions. The avoid register is their externalized memory of failure. It's the system saying "you tried this last week and it was wrong — here's what's correct."

---

#### `src/bootstrap/precompiler.py`
**Why it exists:** Predicts what you'll work on next and pre-compiles context during idle time. When you actually start the session, context is instant if prediction was correct.

**What it must contain:**
- **Next-session prediction:** Analyzes the current phase, sub-task progress, and historical patterns to predict the most likely next actions. "Phase 07 sub-task P7_T3 is 70% complete. Most likely: finish P7_T3 (60% probability), start P7_T4 (30%), user does something unexpected (10%)."
- **Multi-scenario pre-compilation:** Pre-compiles context for the top 3 most likely scenarios. Scenario 1 (finish P7_T3): full context ready. Scenario 2 (start P7_T4): context ready. Scenario 3 (unexpected): fall back to on-demand compilation.
- **Idle time execution:** Runs when the system is idle (no active session). Uses low-priority CPU scheduling. Doesn't interfere with active work.
- **Prediction cache:** Stores pre-compiled contexts with TTL. If user starts a session within the prediction window (default: 24 hours), instant context. If outside window, re-compile on demand.
- **Prediction accuracy tracking:** Tracks how often predictions are correct. "Pre-compiler accuracy: 72% over last 50 sessions." Displays this to the user.
- **Cache invalidation:** If the user makes changes outside the system (manual file edits), invalidates relevant cached pre-compilations. Detected via file modification timestamps.
- **Warm-up on project open:** When you cd into a project directory, the precompiler can detect this and warm up predictions. "Welcome back. I predict you'll continue PHASE_07. Context is ready."

**Why it's necessary:** Every second of waiting for context compilation is friction. This eliminates that friction 72% of the time. Over 200 sessions, that's hours saved.

### SECTION H: MODELS LAYER (Dynamic Multi-Model Support)

This is completely redesigned from my earlier approach. No hardcoded adapters. Fully dynamic model registry.

---

#### `src/models/dynamic_registry.py`
**Why it exists:** A dynamic, file-based model registry that supports ANY model without code changes. Add a model by editing a YAML/JSON file, not Python code.

**What it must contain:**
- **Registry file format:** A YAML (or JSON) file at `~/.cognition/models.yaml` that defines all known models. Structure:
  ```yaml
  models:
    - id: "claude-sonnet-4-20250514"
      provider: "anthropic"
      api_base: "https://api.anthropic.com/v1"
      endpoint: "/messages"
      auth_header: "x-api-key"
      auth_prefix: ""
      capabilities: ["chat", "tool_use", "vision", "extended_thinking"]
      max_context: 200000
      max_output: 4096
      pricing:
        input_per_1k: 0.003
        output_per_1k: 0.015
      tokenizer: "claude"
      default: false
  ```
- **Registry loading:** On system startup, loads the registry file. Validates all entries have required fields. Caches in memory with file watcher — if the YAML file changes, registry auto-reloads.
- **Model discovery:** `registry.list_models()` returns all models. `registry.get_model(model_id)` returns one. `registry.list_by_capability("tool_use")` returns models with tool use.
- **Default model assignment:** User can set a default model per task type: `defaults: heavy_task: "claude-opus-4", light_task: "deepseek-chat"`.
- **Custom model addition:** `registry.add_custom_model(name, api_base, ...)` for adding private endpoints, local Ollama models, or new providers. Stored in user's config, persists across restarts.
- **Capability detection:** On model registration, automatically tests the endpoint with a minimal request to verify: it responds, it supports the claimed capabilities. Flags discrepancies.
- **Provider abstraction:** The registry doesn't care about providers. "anthropic", "openai", "deepseek" are just labels. The request_builder and response_parser handle provider-specific formatting.

**Why it's necessary:** In 6 months, there will be models we haven't heard of today. With a dynamic registry, adding support is editing a YAML file. Without it, adding support is writing 200 lines of Python and hoping it works.

---

#### `src/models/capability_detector.py`
**Why it exists:** Automatically detects what a model can and cannot do. Not all models advertise their capabilities accurately.

**What it must contain:**
- **Automated capability testing:** Sends a series of minimal test requests to determine: does it support system messages? Does it support tool/function calling? Does it support JSON mode? Does it support streaming? What's the actual max context (vs advertised)? What's the actual rate limit?
- **Tool call format detection:** Different providers format tool calls differently. This detects the format automatically by sending a simple tool-use request and analyzing the response structure.
- **Vision capability test:** Sends a tiny base64-encoded image and asks "what color is this?" Verifies the model can process images.
- **Extended thinking test:** For Claude models, tests if extended thinking (reasoning tokens) is supported and what the max thinking budget is.
- **Fallback behavior documentation:** "If this model receives a tool_use request but doesn't support tools, what happens? Does it error? Does it try to emulate tool calling in text?" This determines whether the fallback manager needs to intercept.
- **Performance benchmarking:** Measures latency for a standard prompt (100 tokens input, 50 tokens output). Records p50, p95, p99 latency. Helps the intelligent router decide between equally capable models.
- **Capability report:** Generates a structured capability profile for each model, stored alongside the registry entry. Updated weekly or on manual refresh.

**Why it's necessary:** "Supports tool use" means different things to different providers. This detects reality vs documentation. Prevents sending tool-use requests to models that will error.

---

#### `src/models/intelligent_router.py`
**Why it exists:** Automatically selects the best model for each task. Heavy task → premium model. Light task → cheap model. Respects budget and latency requirements.

**What it must contain:**
- **Task complexity classification:** Analyzes the sub-task to determine complexity. Factors: task_type (BUILD/DEBUG/REFACTOR/TEST), files_affected count, estimated_tokens, whether it involves new code vs modifying existing, language complexity (Rust > Python > JavaScript for difficulty), whether tests exist.
- **Model scoring algorithm:** Each available model is scored on: capability_match (can it do this task?), cost_efficiency (tokens per dollar), latency (ms per token), reliability (error rate from historical data), current_availability (is it rate limited right now?).
- **Cost-based routing:** Simple tasks (write docstrings, format code, write boilerplate tests) route to cheapest capable model. Complex tasks (architecture design, security review, complex refactoring) route to premium models.
- **Budget-aware routing:** If the session is in YELLOW zone, routes to cheaper models even for medium tasks. If in RED zone, routes everything to cheapest model that can handle it.
- **Fallback chain:** Each task has a primary model and 2 fallbacks. If primary is unavailable (rate limited, down), automatically tries fallback1, then fallback2.
- **Parallel routing:** When the orchestra spawns multiple agents, the router can assign different models to different agents simultaneously. Architect agent gets Claude Opus. Test writer agent gets DeepSeek. Both run in parallel.
- **Routing explanation:** Every routing decision is explained: "Assigned claude-opus-4 to ARCHITECT agent (complex design task, high reasoning needed). Assigned deepseek-chat to TEST_WRITER agent (simple boilerplate, cost-optimized). Estimated savings: $2.40 vs using Claude for both."
- **User override:** User can force a specific model: `cc-start --model=gemini-2.5-pro`. Router respects override.

**Why it's necessary:** Using Claude Opus for everything is expensive. Using DeepSeek for everything risks quality on complex tasks. Intelligent routing saves 40-60% on API costs without sacrificing quality.

---

#### `src/models/pricing_tracker.py`
**Why it exists:** Model pricing changes frequently. Tracks current pricing and calculates accurate costs.

**What it must contain:**
- **Current pricing database:** Maintains up-to-date pricing for all registered models. Sources: registry file (user-configurable), periodic API checks (some providers expose pricing endpoints), manual updates.
- **Pricing change detection:** If pricing data is older than 7 days, flags as potentially stale. Compares current pricing against historical to detect significant changes.
- **Multi-format pricing:** Supports different pricing models: per-token (most common), per-request (some image models), per-minute (some real-time models), tiered (first 1M tokens cheap, then expensive).
- **Cost calculator:** `calculate_cost(model_id, input_tokens, output_tokens, reasoning_tokens=0)` returns exact cost in USD. Handles models that charge differently for input vs output vs reasoning tokens.
- **Session cost tracking:** Accumulates cost throughout a session. Every API call updates the running total. The dashboard shows real-time cost.
- **Project cost tracking:** Total cost per project, per phase, per session. Enables "this project has consumed $47.32 in API costs over 23 sessions."
- **Cost projection:** Given a phase's estimated tokens, projects cost for each candidate model. "Phase 08 will cost ~$3.40 on Claude Sonnet or ~$0.42 on DeepSeek."
- **Budget enforcement integration:** Feeds cost data to the proxy's budget_enforcer. When session cost approaches the budget, triggers warnings.

**Why it's necessary:** Cost awareness drives smart decisions. Without it, you discover you spent $200 on API calls when the credit card bill arrives.

---

#### `src/models/fallback_manager.py`
**Why it exists:** Models go down. Rate limits hit. The fallback manager ensures the system keeps working.

**What it must contain:**
- **Health checking:** Periodically (every 60 seconds) checks each provider's health. A simple request to the API status endpoint or a 1-token completion. Tracks: up/down status, current latency, error rate in last 5 minutes.
- **Circuit breaker pattern:** If a model fails 3 times in 60 seconds, the circuit "opens" — the model is temporarily removed from routing (5 minutes). After cooling, one test request is sent. If it succeeds, circuit closes and model returns to routing. If it fails, cooling period doubles (exponential backoff: 5min → 10min → 20min → 40min max).
- **Graceful degradation:** If ALL premium models are unavailable, routes to the best available model with a warning: "Claude Opus is down. Using DeepSeek for design task. Quality may be reduced." System doesn't crash — it adapts.
- **Retry with backoff:** For transient errors (429 rate limit, 503 service unavailable), retries with exponential backoff: 1s → 2s → 4s → 8s → 16s max. For permanent errors (401 unauthorized, 400 bad request), no retry — reports immediately.
- **Rate limit tracking:** Tracks remaining rate limit from response headers. When approaching limit, proactively switches to other models to avoid 429 errors.
- **Fallback logging:** Every fallback event is logged: which model failed, why, which fallback was used, how much extra it cost (if fallback is more expensive). Feeds into reliability metrics.
- **User notification:** "Claude Sonnet is currently degraded (p95 latency: 8.2s vs normal 1.1s). Switched to DeepSeek for this request. Press Ctrl+F to force Claude anyway."

**Why it's necessary:** Without fallbacks, a 5-minute Anthropic outage stops your work. With fallbacks, you might not even notice.

---

#### `src/models/request_builder.py`
**Why it exists:** Every model provider formats requests differently. This normalizes them into a unified format and builds provider-specific requests.

**What it must contain:**
- **Unified request format:** Internally, all requests use a standard format: `{system_prompt, messages[{role, content, tool_calls?}], tools[{name, description, parameters}], max_tokens, temperature, stream}`. This is provider-agnostic.
- **Provider-specific formatting:** Given a unified request and a model_id, builds the actual HTTP request for that provider:
  - Anthropic: `{model, system, messages[{role, content}], tools, max_tokens}`
  - OpenAI: `{model, messages[{role, content}], tools, max_tokens}` (system is a message with role=system)
  - DeepSeek: OpenAI-compatible format
  - Gemini: `{contents[{role, parts[{text}]}], systemInstruction, tools, generationConfig}`
  - OpenRouter: passes through with model name mapping
- **Tool definition translation:** Translates the standard tool format to provider-specific format. OpenAI uses `function` type wrapper. Anthropic uses `tool_use` content blocks. Gemini uses `functionDeclarations`.
- **Thinking/reasoning parameter translation:** Claude's `thinking` parameter → OpenAI's `reasoning_effort` → DeepSeek doesn't support it (stripped with warning).
- **Image handling:** Some providers expect base64 in content blocks. Others expect URLs. The builder normalizes image inputs.
- **Header construction:** Builds appropriate headers per provider: Authorization format, content type, custom headers (Anthropic's `anthropic-version`, OpenAI's `OpenAI-Organization`).
- **Validation before sending:** Checks that the built request doesn't exceed model's max context. Checks that tools are supported. Validates all required fields present.

**Why it's necessary:** Without this, adding a new model means rewriting every component that makes API calls. With this, components use the unified format and the builder handles translation.

#### `src/proxy/api_proxy.py`

### SECTION J: SHIELD LAYER (Hallucination Prevention)

These files validate code before it touches your filesystem. Three stages of increasing depth and cost.

---

#### `src/shield/validation_pipeline.py`
**Why it exists:** Orchestrates the three-stage validation pipeline. Decides which stages to run based on the code change and configured sensitivity.

**What it must contain:**
- **Pipeline orchestration:** Receives proposed code modification (file path, original content, new content). Routes through stages in order: Stage 1 (static, always run, <100ms) → Stage 2 (semantic, run for non-trivial changes, <500ms) → Stage 3 (runtime, run for critical changes, <2s).
- **Sensitivity-based gating:** LOW sensitivity: only Stage 1. MEDIUM: Stage 1 + Stage 2 for files >50 lines. HIGH: all stages for all changes.
- **Early termination:** If Stage 1 blocks the change (critical error), Stages 2 and 3 are skipped. The change is rejected immediately.
- **Parallel stage execution:** Stage 1 and Stage 2 can run in parallel since they analyze different aspects. Results are merged.
- **Result aggregation:** Collects results from all stages. Produces a unified verdict: PASS (no issues), WARN (potential issues, change allowed with warning), BLOCK (critical issue, change rejected).
- **Explanation generation:** When blocking, provides a clear explanation: what was detected, why it's a problem, what the correct approach is. Feeds this to the auto_corrector.
- **Timing tracking:** Records how long each stage takes. If Stage 2 consistently exceeds 500ms, investigates optimization.
- **Pipeline configuration:** Per-language configuration. Python files get AST-based Stage 1. JavaScript files get ESLint-based Stage 1. Configurable per project.

**Why it's necessary:** The pipeline is the gatekeeper. Without it, hallucinated code enters the codebase and causes bugs that take hours to find. With it, hallucinations are caught before they touch disk.

---

#### `src/shield/static_analyzer.py`
**Why it exists:** Stage 1 validation. Fast, syntax-level checks. Catches the most common hallucinations.

**What it must contain:**
- **Syntax validation:** Parses the proposed code with the language's AST parser. If parsing fails, the code has syntax errors — immediately blocked.
- **Import validation:** Extracts all import statements. For each import: checks if the package exists in requirements.txt/package.json. Checks if the imported symbols actually exist in the package (using the truth database). "Import `flask_sqlalchemy` → exists. Import `SQLAlchemy` from it → exists. Import `MagicORM` from it → DOES NOT EXIST → BLOCK."
- **Function call validation:** Extracts all function calls. For each: checks if the function exists in the codebase or installed packages. Checks if the argument count matches the function signature. "Call `authenticate(user, password, use_biometrics=True)` → function exists → signature is `authenticate(user, password)` → too many arguments → BLOCK."
- **Variable usage validation:** Checks that referenced variables are defined. Checks that defined variables are used (warns on unused variables).
- **Type consistency checking:** If the code has type hints, validates type consistency. Passing a `str` to a function expecting `int` → WARN.
- **Indentation and style checking:** Flags inconsistent indentation, mixed tabs/spaces, lines exceeding 120 characters. Not blocking, but warned.
- **Regex-based pattern matching:** Matches against known hallucination patterns: invented library names that look real ("`flask_magic_auth`" doesn't exist), common misspellings of real packages ("`beatifulsoup4`" vs "`beautifulsoup4`").
- **Caching:** Caches validation results per file+content hash. If the same code is validated twice, returns cached result in <1ms.

**Why it's necessary:** 70% of hallucinations are caught at Stage 1. It's fast, cheap, and catches the most obvious errors. Running only Stage 1 (LOW sensitivity) still provides significant protection.

#### `src/shield/semantic_analyzer.py`
**Why it exists:** Stage 2 validation. Deeper analysis of code meaning and consistency. Catches hallucinations that are syntactically correct but semantically wrong.

**What it must contain:**
- **Code-duplicate detection:** Checks if the proposed function/class already exists elsewhere in the codebase (using cosine similarity on embeddings). If a function with 95% similar implementation exists, warns: "This function is nearly identical to `existing_function()` in utils.py. Consider reusing instead of duplicating."
- **Pattern consistency checking:** Compares the proposed code against the project's existing patterns. If the project uses `snake_case` for functions and the proposed code uses `camelCase`, warns. If the project uses async/await and the proposed code is synchronous, warns.
- **Logic error detection:** Simple heuristic checks: conditions that are always true/false, loops with no exit condition, functions with no return statement but type-hinted to return something, unreachable code after return/raise.
- **Security vulnerability detection:** Checks for: hardcoded secrets (API keys, passwords in strings), SQL injection patterns (string formatting in queries), XSS patterns (unescaped user input in HTML), path traversal (user input in file paths).
- **API misuse detection:** Checks if the code uses deprecated APIs. "`datetime.utcnow()` is deprecated in Python 3.12+. Use `datetime.now(datetime.UTC)` instead." Based on the framework versions detected by the scanner.
- **Documentation-code consistency:** If the code includes docstrings/comments, checks if the comment matches the code. A comment saying "Returns a list of users" but the function returns a dict → WARN.
- **Complexity scoring:** Calculates cyclomatic complexity. If a function exceeds threshold (default: 10), warns: "This function has complexity 15 — consider splitting." High complexity correlates with higher hallucination risk.
- **Embedding-based anomaly detection:** Generates an embedding of the proposed code. Compares against embeddings of the existing codebase. If the proposed code is a significant outlier (cosine similarity <0.3 to anything existing), it might be a complete fabrication. Flags for extra scrutiny.

**Why it's necessary:** Stage 1 catches "this function doesn't exist." Stage 2 catches "this function exists but you're using it wrong" and "this code is correct but it's duplicating existing functionality." Catches subtler hallucinations.

#### `src/shield/runtime_sandbox.py`
**Why it exists:** Stage 3 validation. Actually executes the proposed code in an isolated environment. The ultimate hallucination test.

**What it must contain:**
- **Docker-based sandbox:** Spins up a minimal Docker container with the project's dependencies installed. Copies the proposed code into the container. Executes it in isolation. No network access by default (configurable).
- **Test execution:** If the proposed code includes tests, runs them. If tests pass, code is likely correct. If tests fail, the code might be hallucinated.
- **Import resolution test:** Simply tries to import the proposed code. If imports fail (ImportError), the code is hallucinated. This catches Stage 1 false negatives.
- **Smoke test:** Runs the proposed functions with simple inputs. If the function crashes immediately, it's flagged.
- **Side effect monitoring:** Monitors file system changes, network requests, process spawning. Any unexpected side effects are flagged.
- **Resource limits:** CPU: 1 second max. Memory: 256MB max. Disk: 100MB max. Prevents runaway code from affecting the host system.
- **Timeout:** 2 seconds maximum execution time. If code takes longer, it's terminated and flagged as suspicious.
- **Selective activation:** Only runs for critical files: authentication, payment, database, security modules. Marked as "critical" in the project config.
- **Caching results:** If the same code (by hash) was validated before, skips sandbox and returns cached result.

**Why it's necessary:** The ultimate truth test. If the code imports successfully and runs without crashing, it's probably not hallucinated. Catches the most sophisticated hallucinations that pass Stages 1 and 2.

#### `src/shield/truth_database.py`
**Why it exists:** A vector database storing every real symbol in the codebase and installed packages. The source of truth that the validation pipeline queries.

**What it must contain:**
- **Vector embeddings storage:** Uses ChromaDB (or Qdrant) to store embeddings of: every function signature, every class definition, every method, every constant, every importable symbol from installed packages.
- **Semantic search:** Given a function name or description, finds the closest real function. "Find function that handles password hashing" → returns the actual `hash_password()` function with its signature.
- **Fast exact lookup:** Maintains a parallel hash map: function_name → definition_location + signature. O(1) lookup for exact matches. Used by Stage 1 static analyzer.
- **Fuzzy matching:** Uses Levenshtein distance (from algorithms.py) for typo detection. "`authentcate`" → distance 2 from `authenticate` → suggests the real function.
- **Incremental updates:** After every session, updates the truth database with new functions/classes added. Only re-indexes changed files (detected via content hash comparison).
- **Package index:** On project initialization, indexes all installed packages' public APIs. `pip list` → for each package → extract public symbols → store in truth database. This enables import validation.
- **Deprecation tracking:** Tracks deprecated APIs. If code uses a deprecated function, warns with the replacement suggestion.
- **Storage efficiency:** Uses quantization for embeddings to reduce storage. Typical project: 50k-500k symbols → 100MB-1GB vector storage. Acceptable for local use.

**Why it's necessary:** The hallucination shield can only validate against what it knows exists. The truth database is that knowledge. Without it, the shield can only guess.

#### `src/shield/import_validator.py`
**Why it exists:** Specialized validator for import statements. Imports are the most common source of hallucinations.

**What it must contain:**
- **Package existence check:** For each import, verifies the package is in requirements.txt/package.json. If not found, checks if it's a standard library module (using the per-language stdlib list from constants.py).
- **Symbol existence check:** For `from X import Y`, verifies that Y actually exists in package X. Queries the truth database for the symbol.
- **Version compatibility check:** If the package version is known (from lock files), verifies that the imported symbol exists in that version. Some symbols are added/removed between versions.
- **Alias detection:** Detects when the model invents an alias for a real package. "`import numpy as np`" is standard. "`import numpy as numerical_python`" is weird. "`import numpy as pd`" is wrong (pd is pandas). Flags unconventional aliases.
- **Circular import detection:** Detects when the proposed import would create a circular dependency. "File A imports from B, B imports from A" → BLOCK.
- **Import style enforcement:** Matches the project's import style. Absolute imports vs relative imports. Alphabetical ordering. Grouping (stdlib, third-party, local).
- **Auto-correction integration:** When an import is invalid, generates the correction suggestion. "Import `flask.ext.sqlalchemy` is deprecated. Use `flask_sqlalchemy`." Feeds to auto_corrector.

**Why it's necessary:** Import hallucinations are the most common and easiest to catch. A dedicated validator provides fast, accurate import checking without the overhead of full code analysis.

#### `src/shield/signature_checker.py`
**Why it exists:** Validates function/method call signatures. Catches parameter invention.

**What it must contain:**
- **Signature extraction:** From the truth database, retrieves the exact signature of every function called in the proposed code. Signature includes: parameter names, types (if annotated), default values, whether *args/**kwargs are accepted.
- **Argument count validation:** Checks that the number of arguments matches the function signature. Too few (missing required args) → BLOCK. Too many → BLOCK.
- **Keyword argument validation:** Checks that keyword argument names match actual parameter names. "Call `authenticate(user, passcode=...)` → actual parameter is `password`, not `passcode`" → BLOCK with suggestion.
- **Type compatibility checking:** If type hints are present, checks argument types against parameter types. "Function expects `int` but received `str`" → WARN (dynamic languages might work anyway).
- **Required vs optional parameter checking:** Ensures all required parameters (those without defaults) are provided. Optional parameters can be omitted.
- **Deprecated parameter detection:** Flags usage of parameters marked as deprecated in the library's documentation. "`setup(encoding='utf-8')` → parameter 'encoding' is deprecated since v2.0."
- **Context-aware suggestions:** When a signature mismatch is found, generates a helpful error: "`hash_password(password, salt)` — this function accepts `(password, salt, iterations=100000)`. You're missing the required parameter `salt`."

**Why it's necessary:** The model often invents parameter names or adds extra parameters that don't exist. The signature checker catches these before they cause runtime errors.

#### `src/shield/pattern_matcher.py`
**Why it exists:** Matches proposed code against known hallucination patterns from the avoid register. Pattern-based detection.

**What it must contain:**
- **Pattern database:** Maintains a database of known hallucination patterns extracted from past sessions. Patterns are regex + context rules. Example: "Model invents `async_` prefix for synchronous libraries → pattern: `import async_[real_package_name]`."
- **Pattern matching algorithm:** For each proposed code change, runs all relevant patterns. Uses Aho-Corasick algorithm for multi-pattern string matching (efficient for hundreds of patterns).
- **Context-aware matching:** Some patterns only apply in specific contexts. "Model invents Django-style ORM calls in Flask projects" → pattern only activates if project is Flask.
- **Pattern scoring:** Each pattern match has a confidence score. High confidence (>0.9) → BLOCK. Medium (0.5-0.9) → WARN. Low (<0.5) → log for analysis.
- **Pattern learning:** After each session, the knowledge synthesizer analyzes hallucinations and extracts new patterns. These are added to the pattern database automatically.
- **Pattern decay:** Patterns that haven't matched in 50 sessions are archived. Prevents the pattern database from growing unboundedly.
- **Cross-project pattern sharing:** Patterns from one project can be shared with others (via the cross-project transfer module). A hallucination pattern discovered in Project A protects Project B.

**Why it's necessary:** Pattern matching catches hallucinations that structural validation misses. "This import looks valid but we've seen this exact hallucination pattern 14 times before."

#### `src/shield/auto_corrector.py`
**Why it exists:** When the shield blocks a change, doesn't just say "no." Provides a corrected version.

**What it must contain:**
- **Correction generation:** When a hallucination is detected, generates the corrected code. For import errors: replaces hallucinated import with correct import. For signature errors: adjusts arguments to match the real signature. For typo errors: fixes the typo.
- **Confidence-scored corrections:** Each correction has a confidence score. High confidence (>0.95): auto-apply without asking. Medium (0.7-0.95): suggest to user, one-click accept. Low (<0.7): flag but don't suggest auto-fix.
- **Multi-hallucination handling:** If a single code block has multiple hallucinations, generates corrections for all of them. Orders corrections by dependency (fix imports before fixing function calls that use those imports).

- **User feedback integration:** If the user rejects a correction, logs the rejection. If the same correction is rejected 3 times, stops suggesting it.
- **Correction application:** Can apply corrections directly to the proposed code, producing a corrected version that passes validation.
- **Diff generation:** Shows the user exactly what changed: a unified diff of original vs corrected code.

**Why it's necessary:** Blocking without correcting is frustrating. The auto-corrector turns "that's wrong" into "here's the fix." Maintains development velocity.

### SECTION K: ORCHESTRA LAYER (Multi-Agent System)

#### `src/orchestra/conductor.py`
**Why it exists:** The central orchestrator. Decides how many agents to spawn, what each agent works on, and how results are combined.

**What it must contain:**
- **Task decomposition:** Takes a phase or sub-task from the planner. Decomposes into agent-sized work units. "Phase 07: Build authentication" → [Agent 1: Backend implementation, Agent 2: Frontend component, Agent 3: Tests, Agent 4: Documentation].

- **Session management:** The conductor runs within a single session. When the session ends, the conductor saves agent states for the next session to resume.

**Why it's necessary:** A single AI agent working sequentially is slow. Multiple agents in parallel can reduce development time by 40-60%. The conductor makes parallelism manageable.

#### `src/orchestra/agent_spawner.py`
**Why it exists:** Handles the technical details of creating and managing AI agent processes.

**What it must contain:**
- **Agent process creation:** Spawns each agent as an independent process (or thread) with its own API connection, context window, and working directory.
- **Context isolation:** Each agent receives only the context relevant to its sub-task. The backend agent doesn't see frontend files. Prevents context pollution.
- **Resource allocation:** Assigns compute resources per agent: max tokens, max API calls, max execution time. Enforces these limits.
- **Agent identity:** Each agent gets a unique ID, a role description, and a specific prompt that defines its scope. "You are AGENT_BACKEND_01. Your ONLY job is to implement the backend API. Do not modify frontend files."
- **Health monitoring:** Pings each agent every 30 seconds. If an agent doesn't respond within 60 seconds, it's considered stalled. The conductor can restart or reassign its work.
- **Graceful termination:** When an agent completes its task or the session ends, terminates the agent process cleanly. Captures its final output before termination.
- **Agent pool:** Maintains a pool of pre-warmed agents (API connections established, context loaded) for instant task assignment. Reduces spin-up latency.

**Why it's necessary:** Agent management is complex. Without a spawner, you're manually creating and monitoring agents. With it, the conductor just says "spawn 3 agents for these tasks" and it happens.

#### `src/orchestra/task_decomposer.py`
**Why it exists:** Breaks sub-tasks into agent-sized work units. Determines which parts can be parallelized.

**What it must contain:**
- **Dependency analysis:** Analyzes the sub-task to identify independent components. "Build authentication" has independent components: password hashing, token generation, user model. These can be parallelized.
- **Parallelism scoring:** Scores each potential decomposition by: independence_score (how independent are the components?), integration_complexity (how hard to merge results?), speedup_potential (how much faster than sequential?).
- **Agent instruction generation:** For each agent work unit, generates a detailed prompt: what to do, what files to touch, what NOT to touch, expected output format, acceptance criteria.
- **Interface contract definition:** When multiple agents work on interacting components, defines the interface contract upfront. "Agent 1 will create `hash_password(password) -> str`. Agent 2 will create `verify_password(password, hash) -> bool`. Both must agree on the hash format."
- **Work unit sizing:** Ensures each work unit is appropriately sized. Too small (<5k tokens): overhead dominates. Too large (>80k tokens): single agent might not complete. Target: 20-50k tokens per work unit.
- **Fallback to sequential:** If a task cannot be effectively parallelized (components are tightly coupled), recommends sequential execution. Doesn't force parallelism where it doesn't help.

**Why it's necessary:** Bad decomposition leads to agents stepping on each other's toes or producing incompatible outputs. Good decomposition makes parallelism seamless.

---

#### `src/orchestra/protocol.py`
**Why it exists:** Defines the communication protocol between agents. How agents share information and coordinate.

**What it must contain:**
- **Message format:** Structured JSON messages: `{from_agent, to_agent, message_type, payload, timestamp, correlation_id}`. Message types: REQUEST_INFO, PROVIDE_INFO, DEPENDENCY_COMPLETE, CONFLICT_DETECTED, STATUS_UPDATE.
- **Pub/sub message bus:** Agents publish messages to topics (e.g., "authentication_module"). Other agents subscribe to relevant topics. The conductor sees all messages.
- **Synchronous request-response:** For when Agent A needs information from Agent B before proceeding. A sends request, blocks until B responds.
- **Asynchronous event notification:** For when Agent A completes a dependency that Agent B is waiting for. A publishes "hash_password function complete, signature: ..." — B receives and continues.
- **Shared state repository:** A key-value store accessible to all agents. Agents write: "auth_backend_status: complete". Other agents read to check status. Conductor manages access.
- **Conflict notification:** When two agents modify the same file, the protocol detects it and notifies the conflict_resolver.
- **Message persistence:** All messages are logged for debugging. If an agent crashes, its messages can be replayed.

**Why it's necessary:** Without a protocol, agents work in isolation and produce incompatible outputs. With it, they coordinate like a team.

---

#### `src/orchestra/result_merger.py`
**Why it exists:** Combines outputs from multiple agents into a unified result. Handles the integration phase.

**What it must contain:**
- **File-level merging:** When multiple agents produce different files, simply combines them. Agent 1 produced `auth/backend.py`, Agent 2 produced `auth/frontend.js` → merge is straightforward.
- **In-file merging:** When multiple agents modify the same file (different sections), uses a three-way merge algorithm based on the original file. Marks conflicts when agents modify the same lines.
- **Sequential ordering:** Applies agent outputs in dependency order. Agent 1's output (database model) must be applied before Agent 2's output (API that uses the model).
- **Consistency checking:** After merging, runs a consistency check: do all imports resolve? Do all function calls reference real functions? Are there any orphaned references?
- **Test execution:** After merging, runs the test suite (if tests were part of the output). If tests fail, the merge might have introduced bugs.
- **Merge report:** Generates a report: what was merged, conflicts found, conflicts resolved automatically, conflicts requiring manual resolution.

**Why it's necessary:** Parallel agents produce parallel outputs. Without a merger, you're manually combining code. With it, integration is automated.

---

#### `src/orchestra/conflict_resolver.py`
**Why it exists:** Handles conflicts when multiple agents modify the same code in incompatible ways.

**What it must contain:**
- **Conflict detection:** Detects conflicts: same line modified differently, same function modified by two agents, Agent A uses a function that Agent B deleted.
- **Automatic resolution strategies:**
  - Last-write-wins: If Agent B's change clearly supersedes Agent A's.
  - Semantic merging: If agents modified different parts of the same function, combines both changes.
  - Dependency-based: Agent with higher priority (closer to critical path) wins.
  - Majority vote: If 2+ agents agree and 1 disagrees, majority wins.
- **Resolution confidence scoring:** Each automatic resolution has a confidence score. High confidence: apply automatically. Low confidence: ask the user.
- **Manual resolution interface:** When automatic resolution fails, presents the conflict clearly: "Agent 1 changed `login()` to use OAuth. Agent 2 changed `login()` to use JWT. These are incompatible. Choose: [1] OAuth, [2] JWT, [3] Both (combined system)."
- **Conflict prevention:** Analyzes conflict patterns and updates the task decomposer to avoid similar conflicts in the future.

**Why it's necessary:** Conflicts are inevitable in parallel work. Without resolution, they block progress. With automatic resolution, most conflicts are handled invisibly.

#### `src/orchestra/agent_registry.py`
**Why it exists:** Registry of available agent types, their capabilities, and performance history.

**What it must contain:**
- **Agent type definitions:** ARCHITECT (design, planning, no code writing), BACKEND_DEV (backend implementation), FRONTEND_DEV (frontend implementation), SECURITY_REVIEWER (code audit, vulnerability detection), TEST_WRITER (test generation), DOC_WRITER (documentation), REFACTOR (code improvement).
- **Capability matrix:** Which agent types can do which tasks. ARCHITECT can design but shouldn't implement. BACKEND_DEV can implement backend but shouldn't design architecture.
- **Performance history:** Per agent type, per model: average efficiency, hallucination rate, completion rate, user satisfaction score. Used by the conductor to optimize agent-to-task assignment.
- **Agent type templates:** Each agent type has a template prompt that defines its role, constraints, and output format.
- **Custom agent types:** Users can define custom agent types with specific roles. "FRONTEND_REACT_DEV: focuses only on React components, uses hooks, follows our component pattern."

**Why it's necessary:** The conductor needs to know what agents are available and what they're good at. The registry provides this knowledge.

### SECTION L: NAVIGATOR LAYER (Progress Tracking & Recommendations)

---

#### `src/navigator/phase_tracker.py`
**Why it exists:** Tracks phase progress in real-time. Manages state transitions. Answers "how far along are we?"

**What it must contain:**
- **Phase state machine:** Implements the state machine from constants.py. Validates every transition. Prevents impossible transitions.
- **Progress calculation:** Calculates phase completion as a weighted average of sub-task completion. If Phase 07 has 4 sub-tasks weighted by estimated tokens, completion = sum(subtask_progress × subtask_weight) / sum(weights).
- **Sub-task progress tracking:** Monitors which sub-tasks are done, in progress, pending, blocked. Updates in real-time as agents complete work.
- **Bottleneck detection:** Identifies sub-tasks that are falling behind schedule. "P7_T3 was estimated at 30k tokens but has consumed 50k and is only 60% complete. This is a bottleneck."
- **Phase velocity:** Tracks phases completed per week. Projects trend: is velocity increasing (team getting more efficient) or decreasing (project getting harder)?
- **Completion prediction:** Given current velocity, predicts when each phase will complete. "PHASE_07: 2 days remaining. PHASE_08: 5 days. Total project: 45 days."
- **Milestone tracking:** Every 5 phases is a milestone. Tracks milestone completion. Celebrates milestones (positive reinforcement for the user).

**Why it's necessary:** Without progress tracking, you're coding blind. You don't know if you're on schedule or falling behind. The phase tracker provides visibility.

#### `src/navigator/progress_visualizer.py`
**Why it exists:** Generates visual representations of project progress. Terminal-based (ASCII/Unicode) for immediate visibility.

**What it must contain:**
- **Phase progress map:** The 24-phase ASCII art progress bar shown earlier in our discussion. Each phase: status icon, progress bar, completion percentage, dependencies, pointer to current phase.
- **Architecture graph visualization:** Renders the architecture graph as ASCII/Unicode. Components as boxes, dependencies as arrows. Color-coded by status: green (operational), yellow (in development), gray (planned), red (blocked).
- **Token consumption heat map:** A treemap-style display showing which files consumed the most tokens. Each file represented as a proportionally sized block. Color intensity indicates token density. Quickly identifies "this one config file consumed 40% of session tokens."
- **Timeline visualization:** Shows project timeline: past phases (completed, with dates), current phase (in progress), future phases (estimated dates). Highlights the critical path in a different color.
- **Agent utilization display:** Shows which agents were active during the session, their efficiency scores, and whether they completed their tasks.
- **Session dashboard:** Live dashboard showing: session duration, tokens consumed, budget remaining, current cost, efficiency score, active agents, current sub-task, recent hallucinations caught.
- **Export capabilities:** All visualizations can be exported as text (for sharing) or as structured data (for external tools).
- **Color themes:** Configurable color schemes. Accessibility mode (high contrast, no color reliance).

**Why it's necessary:** Humans are visual. A 200-line JSON file of project status means nothing. A color-coded progress map is instantly understood.

#### `src/navigator/dependency_resolver.py`
**Why it exists:** Resolves dependencies between phases, sub-tasks, and components. Answers "what's blocking what?"

**What it must contain:**
- **Dependency graph traversal:** Given any node (phase, sub-task, component), traverses the dependency graph to find: all dependencies, all dependents, the dependency chain depth.
- **Circular dependency detection:** Uses DFS cycle detection from algorithms.py. Reports cycles with clear visualization.
- **Blocker identification:** For any blocked phase, traces back to find the root cause blocker. "PHASE_12 is blocked because PHASE_08 is blocked because PHASE_05 sub-task 3 is incomplete."
- **Unblock order calculation:** Given a blocked phase, calculates the minimum set of actions to unblock it. "To unblock PHASE_12: complete PHASE_08 (2 sessions), then PHASE_10 (1 session)."
- **Impact analysis:** "If we delay PHASE_06 by 1 week, what's the impact?" Traverses forward to find all affected phases and their delay.
- **Parallelism opportunity detection:** Scans the dependency graph for phases that could run in parallel (no dependency relationship). Suggests to the conductor.

**Why it's necessary:** Dependencies are the skeleton of the project. Understanding them is understanding what can be done and in what order.

#### `src/navigator/critical_path.py`
**Why it exists:** Calculates and monitors the critical path — the sequence of phases that determines the minimum project completion time.

**What it must contain:**
- **Critical path calculation:** Uses Dijkstra's algorithm (from algorithms.py) where edge weights are estimated token costs. The critical path is the longest path through the dependency graph — the sequence of phases that takes the most time/resources.
- **Critical path visualization:** Highlights the critical path in the progress map. "These 8 phases are on the critical path. Any delay here delays the entire project."
- **Float calculation:** For non-critical phases, calculates the "float" — how much they can be delayed without affecting the overall timeline. "PHASE_14 has 3 days of float. It's not urgent."
- **Critical path monitoring:** Tracks which phase is currently on the critical path. As phases complete, the critical path may change.
- **Resource allocation recommendation:** Recommends focusing resources (best agents, premium models) on critical path phases. Non-critical phases can use cheaper resources.
- **"What-if" simulation:** "If we add a new dependency from PHASE_10 to PHASE_15, how does the critical path change?"

**Why it's necessary:** Not all delays are equal. A delay on the critical path delays the entire project. A delay on a non-critical phase might not matter. The critical path tells you where to focus.

---

#### `src/navigator/complexity_forecaster.py`
**Why it exists:** Predicts future complexity based on current trends. Warns before complexity spirals out of control.

**What it must contain:**
- **Code complexity metrics:** Tracks: cyclomatic complexity per function, file sizes, import depth, coupling between modules, number of dependencies.
- **Trend projection:** Projects complexity trends forward. "If current growth continues, `auth/backend.py` will reach 3,000 lines in 2 weeks. Files >2,000 lines have 4x the bug density."
- **Complexity hotspots:** Identifies files/modules with the highest complexity. These are the riskiest areas for future changes.
- **Refactoring recommendations:** When complexity exceeds thresholds, recommends refactoring. "`utils.py` has 47 functions across 12 responsibilities. Consider splitting into focused modules."
- **New feature impact on complexity:** Before integrating a new feature, estimates how much complexity it will add. "Adding WebSocket support will increase system complexity by ~15%. Consider if the benefit justifies this."
- **Complexity budget:** Optional feature — sets a complexity budget for the project. When approached, warns and suggests simplifications.

**Why it's necessary:** Complexity creeps up silently. One day everything is clean. Three weeks later, files are bloated and intertwined. The forecaster catches this early.

---

#### `src/navigator/debt_detector.py`
**Why it exists:** Tracks technical debt — TODOs, FIXMEs, skipped tests, deprecated patterns.

**What it must contain:**
- **Code annotation scanning:** Scans the codebase for TODO, FIXME, HACK, XXX, WORKAROUND comments. Tracks: when added (git blame), which session, what it references.
- **Debt aging:** Tracks how long each debt item has existed. Older debt is riskier (context forgotten). "This TODO is 45 days old. The original developer (Session 23) is no longer active on this project."
- **Debt severity scoring:** TODO (minor) → FIXME (bug exists) → HACK (fragile solution) → XXX (dangerous). Higher severity = higher priority.
- **Debt accumulation rate:** Tracks rate of debt creation vs resolution. "Last week: 8 new TODOs, 2 resolved. Net debt: +6. Accumulating faster than resolving."
- **Debt interest calculation:** Estimates the "interest" on technical debt — how much extra effort (tokens) it costs because of messy code. "This HACK in `auth.py` makes every authentication change 30% more expensive."
- **Refactoring ROI calculation:** "Paying off the auth module technical debt would cost 40k tokens but save 15k tokens per session for the next 10 sessions. ROI: positive."

**Why it's necessary:** Technical debt is invisible until it's crippling. The debt detector makes it visible and quantifiable.

---

#### `src/navigator/recommendation_engine.py`
**Why it exists:** Generates actionable recommendations based on all navigator data. Tells the user what to do next.

**What it must contain:**
- **Recommendation generation:** Synthesizes data from phase_tracker, critical_path, complexity_forecaster, debt_detector. Generates prioritized recommendations: "1. Complete PHASE_07 sub-task P7_T3 (critical path, due tomorrow). 2. Pay off auth module technical debt (ROI positive). 3. Split utils.py (approaching complexity threshold)."
- **Priority scoring:** Each recommendation scored by: urgency (deadline proximity), impact (how much it unblocks), effort (estimated tokens), ROI (for refactoring).
- **Recommendation types:** CONTINUE_CURRENT (finish what you started), START_NEXT (begin next phase), UNBLOCK (fix a blocker), PAY_DEBT (refactor), OPTIMIZE (improve efficiency), EXPLORE (research spike).
- **Context-aware recommendations:** Considers what the user just finished. After completing a heavy implementation phase, recommends a lighter task. After a debug session, recommends continuing the original work.
- **User preference learning:** Tracks which recommendations the user accepts vs ignores. Learns their preferences. "You've rejected refactoring recommendations 8 times. I'll stop suggesting refactoring unless critical."
- **Explanation generation:** Every recommendation includes WHY it's recommended. Not just "do X" but "do X because Y is blocked by it and Z depends on it."

**Why it's necessary:** Analysis without action is useless. The recommendation engine closes the loop: analyze → recommend → execute.

### SECTION M: INTEGRATOR LAYER (Handling New Features)

---

#### `src/integrator/feature_classifier.py`
**Why it exists:** When a user wants to add an unplanned feature, classifies what kind of change it is. Determines the integration strategy.

**What it must contain:**
- **Feature type classification:** Analyzes the new feature request against the existing master plan. Classifies as:
  - DEPENDENCY: The feature is required for the current phase to complete.
  - ENHANCEMENT: The feature improves the current phase but isn't required.
  - INDEPENDENT: The feature is unrelated to current work.
  - PARADIGM_SHIFT: The feature fundamentally changes the architecture.
- **Classification algorithm:** Compares the feature description embedding against phase embeddings. If high similarity to current phase → DEPENDENCY or ENHANCEMENT. If high similarity to future phases → schedule accordingly. If low similarity to everything → INDEPENDENT.
- **Novelty detection:** Checks if the feature already exists in the planned features registry. If it's genuinely new → emergent. If it was already planned → remind the user.
- **Scope assessment:** Estimates the scope of the new feature: files affected, tokens required, phases touched.
- **Output:** A `FeatureClassification` object with: type, confidence, rationale, estimated_impact.

**Why it's necessary:** Not all features are equal. Adding a helper function is a DEPENDENCY. Adding real-time collaboration is a PARADIGM_SHIFT. The classification determines the response.

---

#### `src/integrator/impact_analyzer.py`
**Why it exists:** Calculates the impact of adding the new feature on the existing plan. How much disruption will it cause?

**What it must contain:**
- **Disruption scoring:** Multi-factor analysis producing a 0-1 disruption score:
  - Dependency impact (0.3 weight): How many existing components does this touch?
  - Architectural coherence (0.25 weight): Does it fit the architecture or break patterns?
  - Phase shift (0.25 weight): How many phases need to be reordered or added?
  - Rework required (0.2 weight): Does completed work need to be redone?
- **Affected component identification:** Traverses the architecture graph to find all components affected by the new feature.
- **Phase impact analysis:** Identifies which existing phases need modification. Which new phases need creation.
- **Token cost estimation:** Estimates the token cost of integration based on similar integrations from history.
- **Timeline impact:** Estimates how much the project completion date shifts.
- **Conflict detection:** Checks if the new feature conflicts with any planned features. "This WebSocket feature conflicts with the planned polling-based notification system. Resolve before proceeding."
- **Output:** An `ImpactReport` with all findings.

**Why it's necessary:** Without impact analysis, adding a "small feature" can delay the project by weeks. The analysis makes the cost visible before the decision.

---

#### `src/integrator/integration_strategies.py`
**Why it exists:** Implements the 5 integration strategies. Selects the best one based on the feature classification and impact analysis.

**What it must contain:**
- **Strategy 1: Absorb as Sub-task** (disruption 0-0.15). Add the feature as a new sub-task in the current phase. No new phases created. Use when the feature is small and fits current work.
- **Strategy 2: Weave Into Current Phase** (disruption 0.15-0.30). Modify the current phase's scope to include the feature. Adjust sub-tasks, increase budget.
- **Strategy 3: Insert Mini-Phase** (disruption 0.30-0.50). Create a new phase immediately after the current one. Small scope, 1-2 sessions. Existing phases shift down.
- **Strategy 4: Schedule as Parallel Phase** (disruption 0.30-0.60). Create a new phase that runs in parallel with existing phases (if no dependencies). Doesn't delay other work.
- **Strategy 5: Trigger Replanning** (disruption 0.70-1.0). The feature fundamentally changes the project. The master planner regenerates the plan from scratch, preserving completed work.
- **Strategy selection algorithm:** Selects strategy based on disruption score + feature type. DEPENDENCY with low disruption → Absorb. ENHANCEMENT with medium disruption → Mini-Phase. PARADIGM_SHIFT → Replanning.
- **User presentation:** Presents the recommended strategy with rationale, estimated impact, and alternatives. User confirms or chooses different strategy.

**Why it's necessary:** Without strategies, every new feature is a crisis. With strategies, it's a controlled, analyzed decision.

---

#### `src/integrator/graph_updater.py`
**Why it exists:** Updates the architecture graph and dependency graph when new features are integrated.

**What it must contain:**
- **Node addition:** Adds new component nodes to the architecture graph for the new feature.
- **Edge management:** Adds dependency edges between new and existing components. Removes edges that are no longer valid.
- **Graph consistency checking:** After updates, validates the graph: no orphan nodes, no broken references, no circular dependencies.
- **Version tracking:** Graphs are versioned. Each update increments the version. Enables "show me the architecture before we added WebSocket support."
- **Impact propagation:** When a node changes, propagates the change to connected nodes. "Component A was upgraded → Components B and C that depend on A may need updates."
- **Visual diff:** Generates a visual diff of the graph before and after the update.

**Why it's necessary:** The architecture graph is the system's understanding of itself. It must stay accurate as the project evolves.

---

#### `src/integrator/replanner.py`
**Why it exists:** Handles major replanning when a paradigm shift occurs. Preserves completed work while regenerating the plan.

**What it must contain:**
- **State preservation:** Before replanning, saves all completed work, decisions, and architectural knowledge.
- **Plan regeneration:** Calls the master planner with: original goal + new feature requirements + completed work (as constraints).
- **Completed work integration:** The new plan must incorporate all completed phases without redoing them. "Phases 1-7 are complete. The new plan builds on this foundation."
- **Phase remapping:** Maps old phase IDs to new phase IDs. "Old PHASE_08 is now covered by new PHASE_09 and PHASE_10."
- **Budget recalculation:** Recalculates all budgets based on the new plan structure.
- **Impact communication:** Clearly communicates what changed and why. "The plan changed because we added real-time collaboration. 3 new phases added. Completion date moved by 2 weeks."
- **User approval:** Presents the new plan for approval before applying.

**Why it's necessary:** Major changes happen. Without a replanner, the old plan becomes irrelevant and the system loses coherence. With it, the system adapts and stays relevant.

### SECTION N: SIMULATOR LAYER (Mirror World)

#### `src/simulator/mirror_world.py`
**Why it exists:** Simulates sessions before running them. Predicts outcomes without consuming real tokens.

- **Simulation accuracy tracking:** Tracks how accurate simulations are vs actual sessions. Improves the simulation models over time.

**Why it's necessary:** Would you rather know a session will cost $5 before you run it, or after? Simulation enables informed decisions.

#### `src/simulator/code_graph.py`
**Why it exists:** A simplified, probabilistic model of the codebase used for simulation. Not the real code, but a model of it.

**What it must contain:**

- **Change propagation:** If file A is modified, which other files are likely affected? Uses the dependency graph and embedding similarity.
- **Graph maintenance:** Updated after each real session to reflect actual changes. The more sessions, the more accurate the model.

**Why it's necessary:** The simulator needs a model of the codebase to predict impacts. The real codebase is too large for simulation. The code graph is a lightweight proxy.

#### `src/simulator/impact_predictor.py`
**Why it exists:** Predicts which files will be affected by a task before the task runs.

**What it must contain:**

- **Dependency-based propagation:** From primary targets, follows the dependency graph to find secondary targets. "Modifying database.py affects 15 files that import it."
- **Historical pattern matching:** Queries the metrics store for similar past tasks. "Last 3 times we modified authentication, we touched 8 files: [list]. This task is similar."
- **Confidence scoring:** Each predicted file has a confidence: high (will definitely be touched), medium (likely), low (possible).
- **Output:** A ranked list of files with predicted impact probability and estimated lines changed.

**Why it's necessary:** Knowing what files will be touched enables better context compilation and budget estimation.

#### `src/simulator/token_estimator.py`
**Why it exists:** Estimates token consumption for a simulated session.

**What it must contain:**

- **Historical baseline model:** Queries historical data for similar tasks. Returns the distribution of actual token consumption.
- **Kalman filter integration:** Uses the same Kalman filter from algorithms.py as the budget_predictor. Combines feature-based prediction with historical baseline.
- **Confidence interval:** Outputs a range, not a point estimate. "Estimated: 48,500 tokens (90% CI: 38,000-62,000)."
- **Per-phase breakdown:** Estimates tokens per session phase: exploration, implementation, verification.

**Why it's necessary:** Accurate estimation is the foundation of budgeting and simulation. Without it, you're guessing.

#### `src/simulator/risk_assessor.py`
**Why it exists:** Assesses hallucination risk for a simulated session.

**What it must contain:**
- **Risk factor analysis:** Factors that increase hallucination risk: unfamiliar libraries (not in project history), async code, complex algorithms, poorly documented APIs, large files (>1000 lines), multiple interacting systems.
- **Historical risk mapping:** Which files/types of tasks have historically had high hallucination rates?
- **Risk score:** 0-1 score where 0 = low risk, 1 = high risk. Scores >0.5 trigger extra validation in simulation.
- **High-risk area identification:** Identifies specific files or functions that are high-risk. "The payment integration code has historically high hallucination rates. Suggest extra validation."
- **Risk mitigation suggestions:** "High hallucination risk detected. Recommended: use premium model, enable Stage 3 sandbox validation, add extra test coverage."

**Why it's necessary:** Some tasks are riskier than others. Knowing the risk before starting enables proactive mitigation.

#### `src/simulator/alternative_generator.py`

### SECTION O: SYNTHESIZER LAYER (Learning from Experience)

---

#### `src/synthesizer/knowledge_synthesizer.py`
**Why it exists:** Analyzes completed sessions to extract patterns, insights, and lessons. The system's ability to learn.

**What it must contain:**
- **Multi-dimensional analysis:** After each session, analyzes: productivity patterns, hallucination trends, estimation accuracy, agent performance, code quality trends, user behavior patterns.
- **Statistical analysis:** Uses EMA for trends, standard deviation for anomaly detection, correlation analysis for finding relationships.
- **Insight generation:** Produces structured insights: "Sessions between 2-5pm are 23% more efficient than morning sessions." "Hallucination rate on async Python code is 3.2x higher than sync code." "Underestimating token budgets by average 18% — consider increasing buffer."
- **Insight confidence scoring:** Each insight has a confidence based on sample size and statistical significance. Low confidence insights are logged but not acted on.
- **Insight application tracking:** Tracks which insights were applied and their actual impact. "Applied insight #47 (increased budget buffer) — estimation accuracy improved from 82% to 91%."

**Why it's necessary:** Data without analysis is noise. The synthesizer turns raw session data into actionable wisdom.

---

#### `src/synthesizer/pattern_extractor.py`
**Why it exists:** Extracts reusable patterns from successful sessions and hallucination patterns from failures.

**What it must contain:**
- **Success pattern extraction:** Identifies patterns from highly efficient sessions: agent configurations, model choices, task decomposition approaches, context compilation strategies. Codifies these as recommended patterns.
- **Hallucination pattern extraction:** From sessions with high hallucination rates, extracts the specific patterns: "Model invents `async_` variants of synchronous libraries." "Model confuses Flask-SQLAlchemy with Django ORM syntax."
- **Pattern generalization:** Extracts the general rule from specific instances. "3 sessions had import hallucinations for `flask_magic_*` packages → pattern: model invents flask extensions that don't exist."
- **Pattern storage:** Stores patterns in the avoid register (hallucinations) and cross-project transfer database (success patterns).
- **Pattern matching efficiency:** Uses Aho-Corasick for efficient multi-pattern matching against new code.

**Why it's necessary:** Patterns are the difference between "we've seen this before" and "we know to prevent this."

---

#### `src/synthesizer/trend_analyzer.py`
**Why it exists:** Detects trends over time. Is efficiency improving? Is hallucination rate increasing?

**What it must contain:**
- **Time-series trend detection:** Uses EMA and linear regression to detect trends in: token efficiency, hallucination rate, budget accuracy, session duration, code complexity.
- **Trend significance testing:** Not every wiggle is a trend. Uses statistical tests to determine if a change is significant.
- **Trend alerts:** "Hallucination rate has increased from 2% to 8% over the last 5 sessions. This is statistically significant (p<0.05). Investigate."
- **Positive trend reinforcement:** "Efficiency has improved 15% over the last 10 sessions. The RL optimizer's budget adjustments are working."
- **Trend projection:** Projects current trends forward. "If efficiency continues improving at this rate, project will complete 1 week ahead of schedule."

**Why it's necessary:** Point-in-time metrics tell you where you are. Trends tell you where you're going.

**Why it's necessary:** Raw analysis is inaccessible. Insights make it usable.

### SECTION P: OPTIMIZER LAYER (RL Token Allocation)

---

#### `src/optimizer/rl_allocator.py`
**Why it exists:** Uses Reinforcement Learning (Q-learning) to learn optimal token allocation across session phases.

**What it must contain:**
- **State representation:** State = (phase_type, project_size, language, complexity_score). Discretized for the Q-table.
- **Action representation:** Action = (explore_percent, implement_percent, verify_percent) where sum = 100.
- **Reward function:** Reward = efficiency_score (tasks_completed / tokens_used) + budget_adherence_bonus - hallucination_penalty.
- **Q-learning algorithm:** Standard Q-learning with learning rate (α=0.1) and discount factor (γ=0.9). Uses ε-greedy exploration (ε starts at 0.3, decays to 0.05 over 100 sessions).
- **Q-table persistence:** Q-table stored in DNA. Persists across sessions. Grows smarter with every session.
- **Allocation recommendation:** Before each session, the RL allocator recommends the optimal split based on the current state.
- **Continuous learning:** After each session, the actual efficiency is calculated and the Q-table is updated.

**Why it's necessary:** Static 30/50/20 splits are suboptimal. The RL allocator learns what works best for YOUR project, YOUR codebase, YOUR working style.

---

#### `src/optimizer/q_table.py`
**Why it exists:** Manages the Q-table data structure — the core of the RL system.

**What it must contain:**
- **Table structure:** Dictionary mapping state_keys to action_values. state_key = hash(phase_type + size_bucket + language + complexity_bucket). action_values = list of (action, q_value, visits) tuples.
- **State discretization:** Continuous features (project_size, complexity) bucketed into ranges. size: SMALL/MEDIUM/LARGE/HUGE. complexity: LOW/MEDIUM/HIGH.
- **Action discretization:** 21 possible actions (all combinations of explore/implement/verify in 10% increments that sum to 100). Example: (20, 60, 20), (30, 50, 20), (30, 40, 30).
- **Lookup efficiency:** O(1) lookup by state_key. Binary search within action_values for best action.
- **Q-value initialization:** New state-action pairs initialized optimistically (q=0.5) to encourage exploration.
- **Table statistics:** Tracks: total_states, total_actions_explored, convergence_metrics. Indicates how well the Q-table has explored the space.
- **Serialization:** Efficient JSON serialization for storage in DNA. Compression for large tables.

**Why it's necessary:** The Q-table is the RL system's memory. Without efficient storage and lookup, RL is too slow for real-time use.

---

#### `src/optimizer/phase_classifier.py`
**Why it exists:** Classifies phases into types for the RL state representation.

**What it must contain:**
- **Phase type classification:** BUILD (new features), DEBUG (fixing bugs), REFACTOR (improving existing code), EXPLORE (research, no code output), INTEGRATE (connecting systems), OPTIMIZE (performance improvements).
- **Automatic classification:** Analyzes phase description and sub-tasks to classify. "Implement user authentication" → BUILD. "Fix login timeout bug" → DEBUG. "Split monolithic database module" → REFACTOR.
- **Manual override:** User can override classification.
- **Classification confidence:** Some phases are clearly one type. Others are mixed. Provides confidence score.
- **Type-specific defaults:** Each type has default allocation and agent preferences. Used before RL has enough data.

**Why it's necessary:** Accurate state representation requires accurate phase classification. BUILD phases and DEBUG phases need different resource allocations.

---

#### `src/optimizer/reward_calculator.py`
**Why it exists:** Calculates the reward signal for the RL system. The reward determines what the system learns.

**What it must contain:**
- **Multi-factor reward:** Reward = (task_completion_efficiency × 0.4) + (budget_adherence × 0.3) + (code_quality_score × 0.2) + (user_satisfaction × 0.1).
- **Task completion efficiency:** tasks_completed / tokens_used. Higher is better.
- **Budget adherence:** 1.0 if exactly on budget, decreases as you go over or significantly under. Slight under-budget is rewarded (efficiency). Significant under-budget is penalized (overestimated).
- **Code quality score:** Based on: tests passing, no new technical debt, no regression bugs, code review pass.
- **User satisfaction:** Did the user accept the code? Did they have to make many corrections? Lower corrections = higher satisfaction.
- **Normalization:** All factors normalized to 0-1 range before combining.
- **Delayed reward:** Some rewards are only known later (e.g., bug discovered next session). The reward calculator handles delayed rewards.

**Why it's necessary:** The reward function defines what "good" means. A poorly designed reward creates unintended behavior. A well-designed reward creates genuine optimization.

### SECTION Q: PROTOCOL LAYER (Future You)

#### `src/protocol/session_capture.py`
**Why it exists:** At session end, captures everything "current you" knows that "future you" will need.

**What it must contain:**
- **Context snapshot:** Captures: current phase, active sub-task, exact line being worked on, files currently open, recent decisions, assumptions being made.
- **Mental state capture:** Prompts the user with specific questions (from mental_state.py). Captures their answers.

**Why it's necessary:** Tomorrow you is a different person with different context. This captures today's context before it evaporates.

#### `src/protocol/mental_state.py`
**Why it exists:** Asks the right questions at session end to capture critical context.

**What it must contain:**
- **Question bank:** A set of questions that research shows are most valuable for context transfer:
  1. "What's the ONE thing future you absolutely must not forget?"
  2. "What assumption are you making that might be wrong?"
  3. "What would you do differently if you started this session over?"
  4. "What's the simplest next step that would create momentum?"
  5. "What are you confused or uncertain about?"
  6. "Is there anything you're procrastinating on?"
- **Adaptive questioning:** Based on session type, asks different questions. Debug session → "What's your leading theory for the bug?" Build session → "What's the riskiest part of this implementation?"
- **Question rotation:** Rotates questions to avoid survey fatigue. Tracks which questions led to the most valuable future context.
- **Answer quality assessment:** Detects low-effort answers. "Answer too short — future you will need more context. Expand?"
- **Answer storage:** Stores answers in the session capture for compilation into the future prompt.

**Why it's necessary:** Generic context ("worked on auth") is useless. Specific context ("the library's `token_expiry` parameter is actually `expires_in`") saves hours.

#### `src/protocol/decision_logger.py`
**Why it exists:** Logs decisions and their rationale throughout the session, not just at the end.

**What it must contain:**
- **Decision detection:** Automatically detects when the AI makes a significant decision: choosing a library, selecting an architecture pattern, changing approach.
- **Rationale capture:** When a decision is detected, prompts: "Decision: Use Redis for caching. Why? (auto-captured context: need sub-millisecond response times, already using Redis for session storage, team familiar with Redis)."
- **Alternative tracking:** Records alternatives considered and why they were rejected. "Considered: Memcached (simpler but no persistence), in-memory cache (not scalable), database cache (too slow)."
- **Decision graph:** Links related decisions. "Decision to use Redis → led to → Decision to use redis-py library → led to → Decision to use connection pooling."
- **Decision review:** At session end, presents all decisions for user review and correction.

**Why it's necessary:** Code tells you WHAT was decided. Decision logs tell you WHY. When you wonder "why did we use Redis?" six months later, the answer is there.

#### `src/protocol/future_prompt_generator.py`
**Why it exists:** Compiles all captured context into the "Future You" message injected into the next session's bootstrap.

**What it must contain:**
- **Context compilation:** Combines: mental state answers, decision logs, gotchas, unfinished business, session summary.
- **Compression for token efficiency:** The future prompt must be under 300 tokens. Uses aggressive summarization. Prioritizes: gotchas (must not repeat), unfinished business (where to start), key decisions (what changed).
- **Tone adaptation:** The prompt is written in a supportive, helpful tone. "Hey! Here's where we left off. The main thing to know is..."
- **Urgency flagging:** If something is time-sensitive, flags it. "URGENT: The authentication bug is in production. Fix this first."
- **Injection point:** The future prompt is injected into the bootstrap at the beginning, before the strategic context. It's the first thing the AI (and you) sees.
- **Feedback loop:** Tracks whether the future prompt was helpful. "Did this summary help? Y/N." Improves over time.

**Why it's necessary:** This is the handoff between past you and future you. A good handoff saves 15-30 minutes of context re-establishment. A bad handoff leaves future you confused.

### SECTION R: TRANSFER LAYER (Cross-Project Learning)

---

#### `src/transfer/project_registry.py`
**Why it exists:** Maintains a registry of all projects the Cognition Engine has worked on. Enables cross-project learning.

**What it must contain:**

- **Project metadata:** Stores: project name, path, type (web app, CLI tool, library, etc.), language, framework, start date, last active date, total sessions, completion status.

#### `src/transfer/pattern_extractor.py`
**Why it exists:** Extracts successful patterns from completed projects for reuse.

**What it must contain:**
- **Pattern mining:** Analyzes completed project DNA to extract: phase sequences, agent configurations, budget allocations, hallucination patterns encountered, architectural patterns used.
- **Pattern validation:** Only extracts patterns from successful projects (completed with high efficiency scores). Failed projects are analyzed for anti-patterns.
- **Pattern generalization:** Strips project-specific details to make patterns reusable. "Phase 04 in Project A was 'Implement Database Layer' → generalized pattern: 'Database implementation phase for Python web apps takes 3-4 sessions with 2 backend agents.'"

#### `src/transfer/similarity_engine.py`
**Why it exists:** Finds similar projects for pattern transfer.

**What it must contain:**
- **Project embedding:** Generates an embedding for each project based on: project type, language, framework, dependencies, architecture style, size.
- **Similarity search:** Given a new project, finds the top 5 most similar past projects using cosine similarity.
- **Relevance weighting:** Recent projects weighted higher than old projects (technologies change). Same-framework projects weighted higher.
- **Transferability scoring:** How transferable are patterns from Project A to Project B? Based on similarity × pattern generality.
- **Confidence scoring:** Higher confidence when multiple similar projects agree on a pattern.

**Why it's necessary:** Not all past projects are relevant. A pattern from a Rust CLI tool won't help with a React web app. The similarity engine finds relevant patterns.

#### `src/transfer/plan_bootstrapper.py`
**Why it exists:** Uses transferred patterns to bootstrap the master plan for a new project.

**What it must contain:**
- **Plan template generation:** Given a new project's specifications, queries the project_registry for similar projects. Extracts their master plans as templates.
- **Template adaptation:** Adapts the template to the new project. Renames phases, adjusts estimates based on size differences, incorporates new project's specific requirements.
- **Plan merging:** If multiple similar projects exist, merges their plans. Uses the most common phase structure. Resolves conflicts.
- **Estimation calibration:** Uses historical data from similar projects to calibrate token estimates. "Similar auth phases took 120k, 140k, and 135k tokens. Estimate: 132k."
- **Risk prediction:** Predicts risks based on past projects. "In 3 similar projects, the integration phase had the highest hallucination rate. Plan extra validation."
- **User presentation:** Presents the bootstrapped plan. "Based on 3 similar projects, here's a recommended 22-phase plan. You can accept, modify, or start from scratch."

**Why it's necessary:** Starting a new project from scratch is slow. Bootstrapping from past experience gives you an 80% complete plan in seconds.

## 🎯 THE 90% BUDGET GRACEFUL WRAP-UP FEATURE

This is the feature you requested — when the session budget hits 90%, the system gracefully wraps up instead of abruptly stopping.

### How It Works:

```
Session Running
     │
     ▼
Budget at 85% → RED ZONE warning injected
     │
     ▼
Budget at 90% → WRAP-UP PHASE TRIGGERED
     │
     ├── 1. Current API request allowed to complete
     │
     ├── 2. Next request intercept: System prompt injected:
     │    "SYSTEM: Session budget at 90%. Complete your current 
     │     sub-task if possible. Then produce a SESSION HANDOFF:
     │     - What was accomplished this session
     │     - What is partially complete and needs continuation
     │     - Exact file and line to resume from
     │     - Any decisions made and their rationale
     │     - Any new gotchas discovered
     │     - Recommended next sub-task
     │     Do NOT start new sub-tasks. Focus on wrapping up."
     │
     ├── 3. Model receives this, naturally completes current work,
     │    then generates the handoff summary
     │
     ├── 4. Handoff summary captured by operational memory
     │
     ├── 5. At 95% budget: Additional reminder injected
     │
     ├── 6. At 100% budget: All further API calls blocked
     │    Exception: The handoff summary response is allowed
     │
     └── 7. Session end triggered automatically
          Session summary displayed with handoff note
```

### Benefits:
- **No lost work:** Current sub-task completes or reaches a clean stopping point
- **Perfect handoff:** Next session's bootstrap includes the wrap-up summary, so you start exactly where you left off
- **No surprise stops:** The model knows the budget is ending and plans accordingly
- **Budget respect:** Hard stop at 100% prevents overruns
